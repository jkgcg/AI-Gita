What Is It (Plain English)
💡 Before you read this section — six plain-English concepts you need:

LLM (Large Language Model): Software that reads text and generates text in response. Think of it as a very sophisticated autocomplete engine trained on a vast library of human-written text. ChatGPT, Claude, and Gemini are LLMs. You interact with them via an API.

GPU (Graphics Processing Unit): Originally designed for video-game graphics, GPUs are ideal for running LLM mathematics because they perform millions of calculations simultaneously. Running an LLM at scale requires a GPU cluster. Buying and managing these is expensive, specialist work.

Inference: In AI, inference means running the model to get an answer — as opposed to training, which is building the model. When a user sends a message to your chatbot, your system is doing inference. Cloud platforms sell inference as a managed service so you do not need to buy GPUs.

Embedding: A way of converting text into a list of numbers that captures its meaning — so that texts with similar meanings produce similar number lists. Used to power semantic search and RAG systems.

Vector store / Vector database: A database designed to store embeddings and answer: “which stored items are most similar in meaning to this query?” Standard SQL databases cannot do this efficiently. Purpose-built vector databases (pgvector, Pinecone, Weaviate) can.

Chunking: Breaking a long document into smaller pieces before embedding it, because embedding models have a maximum input length and smaller chunks produce more precise similarity matches.

Cloud platforms for AI are the managed infrastructure services that make AI capabilities available at enterprise scale — without requiring you to build GPU clusters, manage model serving infrastructure, or procure frontier model access directly.
  • AWS Bedrock: Amazon's managed LLM API layer, giving access to Anthropic Claude, Meta Llama, Cohere, Mistral, and Amazon Titan models through a unified API, with SageMaker for ML training and deployment
  • Azure OpenAI: Microsoft's managed deployment of OpenAI's models (GPT-4o, GPT-4o-mini, embedding models) in Azure's infrastructure, with enterprise SLAs and EU data residency options
  • Google Cloud Vertex AI: Google's unified ML platform, with access to Gemini models, custom model training, MLOps tools, and a model garden covering third-party models

For architects: choosing a cloud AI platform is partly a capability decision (which models are available, which compliance certifications are held) and partly a lock-in decision. Most enterprises choose primarily on their existing cloud provider relationship and secondarily on AI-specific capability gaps.

Why Should I Care
For Solution Architects

Cloud platform choice affects three concrete things:

  • Which models are available: Azure OpenAI has exclusive enterprise access to GPT-4o with SLAs; AWS Bedrock has Anthropic Claude; GCP Vertex has Gemini. If your use case requires a specific model, the platform follows.
  • Data residency and compliance: Azure OpenAI offers EU-hosted endpoints critical for GDPR-sensitive workloads. AWS Bedrock has region-specific endpoints. Know your data residency requirements before shortlisting platforms.
  • Integration with existing infrastructure: if your data lives in Snowflake on Azure, your pipelines run in Azure Data Factory, and your applications deploy to AKS, Azure OpenAI is lower-friction than cross-cloud connectivity to AWS Bedrock.
For Enterprise Architects

Cloud AI platforms introduce new vendor categories into enterprise procurement:

💡 What is a token? A token is the basic unit of text that LLMs process — and the unit that APIs charge for. Roughly 3–4 characters or 0.75 words in English. “Hello world” = 2 tokens. 1,000 tokens ≈ 750 words ≈ 1.5 pages. APIs charge separately for input tokens (your prompt) and output tokens (the model’s response). When you see pricing like “$2 per million tokens”, that is the cost per million of these text chunks.
  • LLM API endpoints with per-token pricing (new cost structure, different from traditional compute)
  • Managed model serving with availability SLAs (different from self-managed model serving)
  • Data processing agreements (DPA) required for models that process personal data — each cloud provider has a different DPA template

Enterprise architects also need to track: model deprecation policies (Azure OpenAI gives 6–12 months' notice; AWS Bedrock policies vary by model provider), version pinning options, and what happens to your system if a model is retired without a suitable replacement.

Think About It Like This
The Managed Services Analogy — Before cloud computing, companies ran their own data centres: buying servers, maintaining them, managing cooling — all in-house. The cloud's value proposition was: let us manage the infrastructure so you can focus on your applications. Cloud AI platforms apply the same logic to model serving.
  • Azure OpenAI is like a managed database service — you get access to the database engine (GPT-4o) via a standard interface (REST API), in your cloud region, with enterprise SLAs, without managing the database server itself.
  • AWS Bedrock is like a managed message queue (SQS) — a choice of message queue engines (multiple LLM providers) through a unified API, with AWS handling scale, availability, and infrastructure.
  • GCP Vertex AI is the most comprehensive — more like a managed ML platform than just a model API. Google integrates training pipelines, feature stores, model monitoring, and model serving into a unified product. More powerful, but more opinionated about how you structure your ML workflows.
  • The HuggingFace Ecosystem sits alongside all three — a community hub for open-weight models, datasets, and evaluation tools, with managed inference endpoints that work across cloud providers. The model library that all three cloud platforms draw from for their open-weight model support.

4.2 AWS Bedrock: What Architects Need to Know

A managed API layer that gives access to multiple LLM providers (Anthropic, Meta, Cohere, Mistral, Amazon Titan) through a single AWS API, with unified authentication (IAM), logging (CloudWatch), and billing.

Key Architectural Features
💡 What is RAG? RAG (Retrieval-Augmented Generation) means: instead of relying on the model’s built-in training knowledge, you retrieve relevant documents from your own systems at query time and feed them to the model alongside the question. The model answers from your data, not from what it memorised during training. Think of it as giving the model a cheat-sheet of relevant pages from your internal wiki before asking it a question.
  • Inference profiles: pre-configured model deployments with guaranteed throughput — important when you need predictable latency and don't want shared capacity fluctuations
  • Knowledge Bases for Bedrock: managed RAG infrastructure — upload documents to S3, Bedrock handles chunking, embedding, indexing, and a retrieval API layer. Reduces RAG build time; limits flexibility vs custom implementation.
  • Agents for Bedrock: managed agent runtime — define tools (Lambda functions) and orchestration logic; Bedrock handles the ReAct loop (Reason + Act pattern). Faster to set up than LangGraph; less flexible for complex state machines.
  • Guardrails for Bedrock: managed content filtering — topic blocking, PII detection, hate speech filtering. Applied at the API layer before and after model calls.
When Bedrock Is the Right Choice
  • AWS is your primary cloud
  • You need multi-provider model flexibility (today Claude, tomorrow Llama, without infrastructure change)
  • You want to use Claude without direct Anthropic API management (AWS billing, IAM auth)
  • You're building on AWS Lambda / Step Functions and want native integration

Cost note: Bedrock typically adds a small markup over direct provider API pricing. The trade-off is unified billing, AWS SLAs, and AWS-native security and compliance infrastructure.

4.3 Azure OpenAI: What Architects Need to Know

Microsoft's managed deployment of OpenAI's model family within Azure's infrastructure. Unlike direct OpenAI API: runs in your chosen Azure region, authenticates via Azure AD / Managed Identity, logs to Azure Monitor, and comes with Microsoft enterprise SLAs (99.9% uptime).

Key Architectural Features
  • Provisioned throughput: reserve capacity (measured in PTU — Provisioned Throughput Units) for predictable latency and guaranteed throughput. Eliminates shared capacity variance. Critical for production customer-facing applications.
  • Azure AI Content Safety: content moderation as a separate managed service — hate speech, violence, sexual content, self-harm classification. Can be applied to any text, not just model output.
  • Prompt flow: low-code/no-code pipeline builder for LLM workflows — useful for rapid prototyping; less suited for production engineering teams who want code-first control.
  • Azure AI Search: managed vector + keyword search service — integrates directly with Azure OpenAI for RAG patterns. Native hybrid search (vector + BM25 keyword relevance) with managed index.
When Azure OpenAI Is the Right Choice
  • Microsoft enterprise agreement in place
  • GPT-4o specifically required (best reasoning, most tested in enterprise contexts)
  • EU data residency required (strongest compliance story)
  • Azure Active Directory already the identity provider (Managed Identity auth — no API key exposure)
  • Existing Azure data estate (Azure SQL, Azure Blob, Azure Data Factory)
4.4 GCP Vertex AI: What Architects Need to Know

Google's most comprehensive AI platform — not just model API access, but a full ML lifecycle platform including training, feature stores, experiment tracking, model registry, and serving infrastructure, plus API access to Gemini models.

Key Architectural Features
💡 What is the context window? The context window is the maximum amount of text the model can hold in working memory at once — your prompt, conversation history, retrieved documents, and the model’s response all count toward this limit. Think of it as a whiteboard: everything the model needs to reason with must be written on this whiteboard. When it fills up, earlier content falls off and the model can no longer see it. Current models support 128K–1M tokens (roughly 90,000–750,000 words), but long agent workflows with rich tool outputs fill this faster than you expect.
  • Gemini access: Gemini 1.5 Pro (1M context window), Gemini 1.5 Flash (fast, cheap), Gemini 2.0 Pro. Gemini 1.5 Pro's 1M token context window enables full-codebase or full-document-library analysis without chunking.
  • Model Garden: curated collection of open-weight and third-party models (Llama, Mistral, Falcon, Gemma) available as managed endpoints.
  • Feature Store: managed feature store for ML — centralises feature computation, storage, and serving. Reduces train-serve skew.
  • AutoML: automated model training for tabular, text, image, and video tasks — low-code option for non-ML-engineer teams.
  • BigQuery ML: run ML models directly from BigQuery SQL — useful for data teams without Python ML expertise.
When Vertex AI Is the Right Choice
  • GCP is your primary cloud
  • Gemini's long context window (1M tokens) is required for your use case
  • You have a significant traditional ML workload alongside LLM use cases (Vertex's MLOps tooling is more comprehensive than Bedrock's)
  • BigQuery is your primary data warehouse (tight integration)
  • Google Workspace is your productivity suite (Gemini integrations across Docs, Sheets, Gmail)

Cloud platforms add a fourth option to the Build vs Buy vs Fine-tune framework: Configure.

4.6 Technical Debt in AI Systems
Note: If you are currently evaluating which platform to choose, complete Sections 4.1–4.5 first, then return here for the long-term view.

AI systems accumulate technical debt faster than traditional software systems because:

Sources of AI Technical Debt
  • Model dependencies: code written for GPT-4 may need updating for GPT-4o; model updates are more frequent than database version updates
  • Prompt drift: prompts that worked well 6 months ago may perform worse on newer model versions. Prompts without version control and evaluation gates accumulate silent quality debt.
  • Data pipeline debt: RAG systems are only as good as their knowledge base. Stale documents, inconsistent chunking, poor metadata degrade retrieval quality over time without visible failures.
  • Evaluation debt: AI systems without evaluation harnesses have no reliable way to detect quality regressions. The longer you operate without it, the more production issues trace back to undetected model changes.
  • Governance debt: responsible AI controls added after deployment are much harder than those designed in. PII handling, audit logging, content moderation added retrospectively require touching every AI call in the codebase.
Debt Reduction Strategy

Quarterly AI system reviews covering:

  • Model version currency — are you on a deprecated version?
  • Evaluation harness completeness — are quality regressions detectable?
  • Knowledge base freshness — are indexed documents still current?
  • Governance control coverage — are all PII handling, audit logging, and content moderation controls in place?
  • Cost optimisation — have model pricing changes created better options?

Check Yourself (3–5 Questions)
Self-Assessment Questions
  1. Your legal team requires that all AI processing of customer personal data occurs within the EU. Which cloud platform options satisfy this and what must be verified?

    All three major platforms have EU regions, but "EU region" is not sufficient on its own. Verify: (1) the specific AI service (not just the cloud platform) has EU-region endpoints; (2) the DPA is signed with the provider — the contractual basis under GDPR Article 28; (3) cross-region data transfer is explicitly disabled in your API configuration; (4) sub-processors: if the cloud provider uses sub-processors for the AI service, are those sub-processors also EU-compliant?; (5) logging and monitoring data also stays in the EU region. Azure OpenAI has the strongest story for European enterprises because the entire stack (compute, networking, logging, identity) stays within Azure's EU infrastructure under a Microsoft enterprise agreement.

  2. A developer proposes using AWS Bedrock's "Knowledge Bases" managed RAG service to avoid building a custom RAG pipeline. What are the advantages and trade-offs?

    Advantages: speed to deployment (hours vs weeks), maintenance reduction (no vector database to operate), native AWS integration, and iterative improvement without re-engineering. Trade-offs: (1) chunking control — managed services use opaque chunking strategies; custom content structure may require custom chunking for quality retrieval; (2) retrieval customisation — cannot implement advanced patterns like ColBERT, HyDE, or custom reranking without custom code; (3) vendor lock-in — migrating to a custom stack is significant re-engineering; (4) observability — limited per-request retrieval debugging when retrieval quality is poor on specific queries. Recommendation: use managed services for initial deployment and low-to-medium complexity use cases. Design the application layer to call the RAG endpoint via a clean interface abstraction so the underlying implementation can be swapped if needed.

  3. An architect says "standardise on GCP Vertex AI because it has the most comprehensive MLOps tooling." A developer counters "but we don't need MLOps — we're just calling LLM APIs." Who is right?

    Both are partially right. The developer is correct that for simple LLM API call patterns, traditional MLOps tooling (training pipelines, feature stores, model registries) is largely irrelevant. The architect is correct that as AI systems mature, MLOps-adjacent concerns become real: evaluation harnesses need to run on a schedule; model version tracking and regression testing apply to prompt + model version combinations; any fine-tuning planned makes a model registry and experiment tracking valuable. The resolution: evaluate on your actual requirements — which models do you need, what are your data residency requirements, what is your existing cloud commitment. MLOps tooling breadth is one factor, not the deciding one for LLM-API-first teams.

  4. Your organisation runs on AWS. A business unit wants to use Claude Opus for complex legal document analysis. How do you provide access and what governance controls do you implement?

    Access via AWS Bedrock — provides managed Claude access through IAM authentication, consolidated billing, and AWS-native logging. Steps: (1) enable Bedrock in the relevant AWS region (eu-west-1 or eu-central-1 for EU data residency); (2) create an IAM role with least-privilege Bedrock invoke permissions scoped to Claude Opus model IDs only; (3) configure VPC endpoints for Bedrock if the application runs in a private VPC; (4) enable CloudWatch logging for all Bedrock API calls. Governance controls: data classification check (legal documents likely contain sensitive information — confirm DPA covers this category and endpoint stays in required region); model version pinning (specify exact model version to prevent unexpected behaviour from provider-side updates); access logging (all invocations logged with the IAM principal for audit trail); cost controls (set AWS Budget alerts — Opus is expensive; a runaway loop would be costly); output governance (legal analysis is high-stakes — define the human review gate explicitly).

  5. What is "provisioned throughput" in Azure OpenAI or AWS Bedrock, and when should an architecture require it vs using shared capacity?

    Provisioned throughput (Azure: PTU — Provisioned Throughput Units; Bedrock: Provisioned Throughput) is a reserved compute allocation — you pay for dedicated capacity guaranteeing a specified number of tokens per minute, rather than sharing capacity with other customers. Shared capacity means your requests compete with all other customers' requests, causing during peak periods: rate limit errors (429s) requiring retry logic, latency spikes, and unpredictable performance. Provisioned throughput eliminates these. When to require it: (1) customer-facing applications with latency SLAs; (2) high-volume predictable workloads where provisioned is typically cheaper at that volume; (3) production applications where rate limit errors are unacceptable. When shared is fine: (1) internal tools, batch workloads, non-real-time use cases; (2) low-volume or bursty workloads where reserved capacity would be idle; (3) development and testing. Cost: provisioned throughput requires minimum commitments (typically monthly) regardless of actual usage — it's a fixed cost. Only makes sense when utilisation will be high enough to justify the reservation.

8.1 SageMaker Pipelines: ML Orchestration on AWS

For organisations running traditional ML (not just LLM APIs), SageMaker Pipelines provides a managed DAG orchestrator for ML workflows:

SageMaker Pipeline components: ├── Processing Step: data preprocessing (Spark or Python) ├── Training Step: model training on managed compute ├── Evaluation Step: automated metrics computation ├── Condition Step: branch on evaluation results ├── Register Step: push to Model Registry if quality passes └── Deploy Step: update endpoint if approved
Each step runs in a managed container on transient compute — no persistent infrastructure required. The pipeline is versioned in SageMaker; execution history, logs, and artifacts are tracked automatically. For LLM workflows, SageMaker Pipelines can orchestrate: dataset preparation, fine-tuning jobs, evaluation harness execution, and model registration.
8.2 HuggingFace Ecosystem: The Model Library

HuggingFace is the de facto library and hosting platform for open-weight models:

Core Components
  • Model Hub: 500,000+ models available for download — base models, fine-tuned variants, embedding models, classification models
  • Datasets: 100,000+ curated datasets for training and evaluation
  • Transformers library: the standard Python library for loading and running models
  • PEFT library: standard implementation of LoRA, QLoRA, and other parameter-efficient fine-tuning methods
  • Evaluate library: standardised evaluation metrics
  • Inference Endpoints: managed GPU hosting for your chosen HuggingFace model — no self-managed GPU required, pay-per-second
Architect Perspective

HuggingFace is the source for most open-weight models regardless of which cloud you ultimately host them on. When evaluating an open-weight model for fine-tuning or self-hosting, the model card on HuggingFace is the authoritative documentation for:

  • Base model and training data
  • Evaluation scores
  • Intended use and limitations
  • Licence terms
Key Takeaways
  • Cloud platform selection is primarily driven by existing cloud commitment and data residency requirements — not AI capability differences. Azure OpenAI for Microsoft shops with EU requirements, AWS Bedrock for AWS-first organisations, GCP Vertex for Google-heavy environments. The capability differences between platforms narrow quarterly; vendor integration and compliance are more durable differentiators.
  • "EU region" is necessary but not sufficient for GDPR compliance — verify the DPA, the specific AI service's region availability, and that cross-region data transfer is disabled. A well-intentioned EU region selection can be undermined by default cross-region logging, sub-processor chains that include US entities, or an unsigned DPA.
  • Managed RAG services (Bedrock Knowledge Bases, Azure AI Search integration) are the right starting point, not a long-term compromise — but design clean interfaces so you can migrate. Speed to deployment and reduced ops burden justify managed services. Design the application layer with an abstraction over the RAG endpoint, so the underlying implementation can be swapped without rewriting the application.
  • Provisioned throughput is mandatory for customer-facing AI applications, optional for internal tools. Shared capacity rate limits and latency spikes are acceptable for internal tools where occasional delays are tolerable. They are not acceptable for customer-facing applications where P95 latency is part of the SLA. Provision capacity before launch, not after the first customer complaint.
  • AI technical debt accumulates faster than traditional software debt — schedule quarterly AI system reviews. Model deprecation, prompt drift, stale knowledge bases, and unevaluated governance controls compound silently. A quarterly review covering model version currency, evaluation harness completeness, knowledge base freshness, and cost optimisation opportunities keeps the system from degrading invisibly.
