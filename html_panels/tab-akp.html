What Is an Agentic Platform?

What is an LLM? An LLM (Large Language Model) is a software service — accessible via an API — that takes text as input and produces text as output. It has no database connection, no memory between calls, and no ability to take actions in external systems by default. You call it like any other API: send a request (your prompt), receive a response (the generated text). GPT-4o, Claude, and Gemini are examples. That is the entirety of what an LLM is on its own.

What this module is about: An Agentic Knowledge Platform is the enterprise infrastructure layer that transforms an LLM — which on its own can only read text and write text — into a system that can take actions, access your data, coordinate with other AI agents, remember past interactions, and do all of this safely under your governance and compliance framework.

A single LLM is like a very smart consultant who can only answer questions by talking — no phone, no laptop, no access to your systems, no memory of the last meeting. An AI agent gives that consultant a phone, a laptop, access to your systems, and a notebook. It can now look things up, run calculations, call APIs, read emails, write files, and remember what happened last week.

An Agentic Platform is the enterprise infrastructure that makes this safe, manageable, and scalable — providing a shared model gateway, a tool registry, a memory system, a governance layer, and an orchestration runtime.

Why Should I Care
For Solution Architects

The moment AI takes actions (not just answers), the design questions change completely:

  • Authorization: which systems can the agent call, with what permissions, on behalf of whom?
  • Blast radius: if the agent makes a mistake, what is the maximum damage? A hallucinated summary is recoverable; an erroneous order cancellation is not.
  • Observability: how do you trace what an agent did and why — across 12 tool calls that happened in 8 seconds?
  • Cost control: an agent that loops unexpectedly can make 500 LLM calls in a minute. Without budget caps, that is an unplanned bill.

An agentic platform answers all of these with infrastructure rather than per-project point solutions.

For Enterprise Architects

Agentic systems are the first AI pattern that crosses your entire integration estate. A single agent may touch your CRM, OMS, ERP, knowledge base, email system, and analytics platform — all in one workflow.

The Agent Knowledge Platform (AKP) uses a 6-plane model to decompose this complexity into governable layers. Each plane maps to something you already have: identity, API gateway, message bus, data platform, knowledge store, and observability stack. The AKP organises how AI agents interact with them — it does not replace them.

Think About It Like This — The Enterprise Help Desk
Imagine running a large enterprise help desk for a retail company. You have employees (agents) who handle customer issues:
  • The phone switchboard (Model Gateway): Every incoming call goes through the switchboard. It routes calls to the right employee, logs the call, and enforces policies. No employee has a direct line that bypasses it.
  • The tool cabinet (Tool Registry): Employees can look up orders, issue refunds, schedule callbacks, and escalate tickets. But not every employee can issue a refund — there is a permissions matrix. The cabinet is locked; employees sign out tools based on their role.
  • The case notes system (Memory): When a customer calls back, the employee can pull up notes from the last call (episodic), search the knowledge base for known issues (semantic), and see what they have already tried this call (working).
  • The supervisor layer (Governance): Refunds over £500 require sign-off before they go through. There is an audit log of every action taken.
  • The call routing engine (Orchestration Runtime): Manages the queue, assigns calls, handles escalations, retries dropped calls, and ensures no single employee gets overwhelmed.
Key insight: you already know how to build this infrastructure. API gateways, identity systems, audit logs, RBAC, message queues — these are not new problems. The AKP applies your existing enterprise patterns to AI agents.
Step-by-Step — The AKP Six-Plane Architecture
Start here: an Agent Core is a loop — Think (LLM call) → Decide (tool needed?) → Act (tool call) → Observe (add result to context) → Repeat. This is the ReAct pattern (Reasoning + Acting). Everything else is infrastructure to make this loop safe, governed, and observable at scale.
Plane 1 — Surfaces
User-facing entry points: chat interfaces, API endpoints, embedded widgets, voice channels, CLI tools. Each surface authenticates the caller and translates the request into a standard agent invocation.
Plane 2 — Agent Runtime
Execution environment for the Agent Core loop. Manages the lifecycle of agent tasks: spawning, scheduling, timeout handling, retries, and parallelism. Coordinates multiple concurrent agents (manager delegating to specialists).
Plane 3 — Tool Plane
Managed registry of capabilities agents can invoke. Each tool entry specifies: what the tool does, what parameters it accepts, who is authorized to call it, rate limits, and how to handle errors. Includes REST API calls, database queries, code execution, browser automation, email/calendar integrations.
Plane 4 — Knowledge Plane
Memory and retrieval layer: vector stores for semantic search, document stores for raw retrieval, graph databases for relationship-aware queries, caches for frequently accessed context. Allows agents to "know things" beyond training data and the current conversation.
Plane 5 — Governance Plane
Policy enforcement, guardrails, audit logging, budget management, compliance controls. Every LLM call and every tool call passes through governance before execution: PII redaction, content filters, action whitelisting, budget caps, GDPR handling, EU AI Act transparency requirements.
Plane 6 — Data Backbone
Underlying data infrastructure: message queues, event streams, data pipelines connecting the agent platform to your enterprise data systems. Ensures agents work from current, consistent data — not stale snapshots.
Enterprise Example — Customer Return Resolution
Scenario: Retail Company — Missing Item Refund

Customer says: "I ordered 3 pairs of shoes last week, two arrived but one is missing from the box. I want a refund."

Without an agentic platform: support agent manually logs into OMS, checks the manifest, cross-references the warehouse scan log, checks return policy, creates a refund ticket, emails the customer. 12–20 minutes of work.

With an agentic platform — 6 tool calls, ~15 seconds, zero human involvement:

1. OMS.get_customer_orders(customer_id) → identifies the relevant order 2. OMS.get_order_details(order_id) → 3-item order, all shipped as one parcel 3. warehouse.get_scan_log(parcel_id) → weight discrepancy: 2.1 kg dispatched vs 3.2 kg expected 4. policy_engine.get_return_policy(item_category="footwear") → eligible for refund without return 5. OMS.create_refund(order_id, item_id, amount=79.99) → refund created 6. comms.send_email(customer_id, template="missing_item_refund") → confirmation sent

Governance controls in this flow: refunds over £200 require human approval; every tool call is in the audit log with LLM reasoning trace; customer PII is redacted in LLM calls (agent works with IDs); budget cap halts agent if it exceeds 20 LLM calls on a single ticket.

What you design as an architect: the tool registry, the authorization matrix, the escalation thresholds, and the observability schema.

Check Yourself — 5 Questions
Self-Assessment

Use these to verify you can reason through the architecture — not just recall definitions.

  1. Your engineering team proposes giving a customer service agent direct database write access "because it's simpler." What is the architectural objection? — Direct DB write bypasses authorization, audit, and governance controls the tool registry enforces. If the agent miscalculates a refund or receives malformed LLM output, it can corrupt records with no audit trail and no human intervention point.
  2. An agent designed for 1 LLM call per ticket makes 400 calls in a minute. What platform controls should prevent this? — Per-agent budget caps in governance; loop detection in the runtime (same tool, same args repeatedly); task-level timeout; rate limits on the model gateway.
  3. What is MCP and why does it matter for enterprise architecture? — MCP is an open standard interface for how agents discover and invoke tools. It enables a shared tool library: your OMS integration, logistics wrapper, and knowledge base search are built once, registered centrally, and available to all agents — rather than each team rebuilding the same integrations independently.
  4. Your legal team asks: "Can we trace why an agent made a wrong decision that led to an incorrect charge?" — The audit log must capture: the full LLM input, the LLM reasoning output (the "thought" before the tool call), the specific tool call with parameters, the tool response, and the final decision output. OTel extended with GenAI semantic conventions is the current production standard.
  5. How does error handling differ between a traditional API integration and an agentic platform? — In traditional integration, errors are explicit exceptions. In agent systems, the LLM may silently absorb a tool error, continue, and produce a confidently wrong answer. Operations teams must watch for behavioral anomalies (unexpected tool sequences, budget spikes, quality drops), not just error codes.

Plane 7 — Evaluation Plane
Continuous evaluation infrastructure: LLM judges that assess quality of other LLM outputs, golden dataset tests run on every deployment, A/B frameworks for comparing agent versions. In production you cannot manually review every interaction — the evaluation plane automates quality assurance.
Plane 8 — Persona / Identity Plane
Agent identity management: each agent has a defined persona (name, tone, capability scope, escalation behavior), role-based access profile, and versioned configuration. Manages rollouts — routing 10% of traffic to a new agent version while the evaluation plane monitors quality.
Context Budget Management

As the agent loops, its context grows. A 128K window can fill up in a long workflow. When it fills, the agent must truncate (losing information) or summarize (lossy compression). Cost is proportional to context size.

💡 What is a token? A token is the basic unit of text that LLMs process — and the unit that APIs charge for. Roughly 3–4 characters or 0.75 words in English. “Hello world” = 2 tokens. 1,000 tokens ≈ 750 words ≈ 1.5 pages. APIs charge separately for input tokens (your prompt) and output tokens (the model’s response). When you see pricing like “$2 per million tokens”, that is the cost per million of these text chunks.
  • Sliding window: keep the most recent N turns, drop older ones
  • Summarization: call the LLM to compress history when context exceeds a threshold
  • Structured working memory: store data as compact JSON objects rather than raw conversation text — much more token-efficient
  • Episodic memory: after a task completes, extract key facts to durable memory and clear working context for the next task
Prompt Drift Detection

In a long-running agentic deployment, the effective prompt changes over time because: few-shot examples become dated; tool descriptions reference APIs that have been updated; the system prompt was written for a model that the model gateway has since rotated away from.

Detection pattern: periodically replay a golden set of test inputs and compare the agent's output to a reference baseline. If output drift exceeds a threshold, trigger a review. This is part of the evaluation plane.

IIB Adapter Pattern — Legacy Integration

When integrating with legacy enterprise systems (SAP, IBM MQ, COBOL mainframes) that lack modern REST APIs, use integration middleware as the translation boundary:

Agent Tool Call (JSON/REST) ↓ Integration Middleware (MuleSoft / IIB / Azure Logic Apps) → translates REST → SOAP / flat file / MQ message ↓ Legacy System (SAP, mainframe)

The tool registry wraps the middleware endpoint as if it were a native API. The agent has no knowledge of the underlying legacy protocol — it calls get_inventory_levels(), the middleware handles the rest.

Key Takeaways
  • An agent is just an LLM with tools and a loop. The Agent Core is: think (LLM call) → act (tool call) → observe (add result to context) → repeat until done. Everything else is infrastructure to make this loop safe, governed, and observable at enterprise scale.
  • The Model Gateway and Tool Registry are your primary governance controls. Every LLM call goes through the gateway (auth, cost tracking, routing, rate limiting). Every tool call goes through the registry (auth, blast radius controls, audit log). Without these, there is no consistent governance — each agent team manages its own.
  • Agentic systems require a different audit model. Traditional audit captures what happened. Agentic audit must capture why — the LLM reasoning trace that preceded each action. OpenTelemetry extended to LLM spans is the current production standard.
  • Blast radius management is the key design principle for tool authorization. Read-only tools need minimal controls. Tools that write to external systems, send communications, or make financial changes need: amount limits, human approval thresholds, loop detection, and irreversibility flags. Design the permission matrix before you design the agents.
  • The AKP is not a new architecture — it is your existing enterprise stack extended for AI agents. Your API gateway, IAM, message bus, data platform, and observability stack are all reused. The AKP adds the AI-specific layer on top: model gateway, tool registry, knowledge plane, and governance controls that understand LLM-specific failure modes.
