What Is It (Plain English)
🎯 Explain Like I'm an Architect: You've shipped traditional software before. Once your Java service is deployed, it doesn't suddenly start giving wrong answers because the world changed outside it. AI models are different — a fraud detection model trained on pre-COVID spending patterns started flagging legitimate lockdown purchases as fraud, because human behaviour changed overnight but the model didn't know. MLOps is the discipline that (a) automates getting models from experiment to production reliably, and (b) keeps watching them after they're live so you catch that kind of silent degradation before users do.
MLOps (Machine Learning Operations) is the discipline of reliably moving AI models from experiment to production — and keeping them working once they're there.
LLMOps is the specialised variant for Large Language Models and generative AI systems. Same goals, different tools, different failure modes.
If you've worked with DevOps, the analogy is direct: DevOps brought CI/CD, monitoring, and infrastructure-as-code to software delivery. MLOps does the same for AI models. The difference is that AI models are not just code — they encode learned behaviour from data, and that behaviour can degrade silently as the world changes around them.
⚠ Common Misconception: "MLOps is just DevOps with a GPU." Not quite. DevOps assumes that if your tests pass, your code behaves correctly — forever, until you change the code. ML systems can fail without any code change at all. The inputs change, the world changes, and a model that was accurate last month becomes inaccurate this month. MLOps adds an entirely new monitoring concern (drift detection) that has no equivalent in standard DevOps.
What MLOps covers in practice:
  • The ML pipeline: automated workflows for data ingestion, feature engineering, model training, evaluation, and deployment
  • Model registry: versioned storage of trained models, with metadata about how they were trained and what they were trained on
  • CI/CD for ML: automated gates that run tests, evaluations, and quality checks before any model reaches production
  • Drift detection: continuous monitoring to catch when a model's real-world performance has degraded since deployment
  • Feedback loops: capturing production outcomes to feed back into future training data
LLMOps adds on top of this:
  • Prompt management: versioning, testing, and deploying prompts as first-class artifacts (not hardcoded strings)
  • LLM evaluation: assessing quality, safety, and relevance using LLM judges and benchmark suites
  • Cost tracking: per-model, per-application, per-team token usage
  • Observability for non-deterministic outputs: monitoring quality signals, not just error rates
The core principle: production is where AI systems actually fail. MLOps is the engineering discipline that catches those failures before users do — and builds the infrastructure to fix them fast.
Why Should I Care
For Solution Architects

Every AI system you design will eventually face the same operational questions:

  • How do you know the model is still working well six months after launch? The model doesn't send an error when it starts giving worse answers. You need monitoring that detects quality degradation, not just system errors.
  • How do you update a prompt or swap a model without a production incident? Without MLOps, prompt changes are risky manual operations. With MLOps, they're versioned, tested, staged, and rolled back just like code.
  • Who owns the AI system's ongoing quality? In traditional software, QA owns test coverage. In AI systems, quality ownership is ambiguous — and MLOps defines it.

The decisions you make in architecture — which models to use, how to integrate them, what SLAs to commit to — all have downstream MLOps implications.

For Enterprise Architects

MLOps is where AI strategy meets enterprise governance. The questions that matter at the enterprise level:

  • Reproducibility: if an AI system made a decision last month, can you recreate exactly the model state, data, and inputs that produced it? GDPR Article 22 requires explainability for automated decisions; the EU AI Act requires documentation of high-risk AI systems; SR 11-7 (US financial services) mandates model risk management including audit trails.
  • Cost governance: LLM API costs are variable and can spike unexpectedly. Without per-team cost tracking and budget controls, AI costs are invisible until the invoice arrives.
  • Model lineage: which data was used to train this model? Was any of it PII? Was it from a licensed source? This is a compliance question — and MLOps tooling is where the answer lives.
Think About It Like This
The Pharmaceutical Manufacturing Analogy

Imagine a pharmaceutical company developing a new drug. The research team (data scientists) discovers the compound works in the lab (model training). But getting from lab discovery to patients in pharmacies requires an entirely separate discipline: pharmaceutical manufacturing and quality assurance.

The manufacturing process (ML pipeline) is the automated, reproducible sequence: raw ingredients (data) → processing (feature engineering) → synthesis (training) → quality testing (evaluation) → packaging (model artifact) → distribution (deployment). Every step is documented. The same inputs always produce the same outputs.

Batch certification (CI/CD gates) means no drug batch ships without passing a full quality test suite. The tests are automated and run on every batch. A batch that fails a test is quarantined — not manually approved by the researcher who made it.

Pharmacovigilance (drift detection) is the post-market surveillance system. After the drug is in the market, the company monitors for unexpected side effects, adverse events, and efficacy changes in the real-world population.

The batch record (model registry + lineage) is the complete, immutable record of exactly what went into each batch: ingredient sources, processing steps, equipment IDs, test results. Required for regulatory compliance and for investigating adverse events.

Replace "drug" with "model," "patients" with "users," and "adverse events" with "quality degradation" — and you have MLOps.

4.1 The ML Pipeline Architecture
🎯 Explain Like I'm an Architect: Think of a car assembly line. Every car goes through the same sequence of stations in the same order — welding, painting, engine fitting, quality checks — and no car skips a station. An ML pipeline is that assembly line for AI models. Raw data goes in one end; a tested, packaged, deployable model comes out the other. The key property: it's automated and reproducible. If you run it again with the same data, you get the same model. That reproducibility is what makes debugging and auditing possible.
What it is
An ML pipeline is the automated sequence that takes raw data and produces a deployable model artifact. The key property: every step is automated, versioned, and reproducible.

4.2 What LLMOps Changes
🎯 Explain Like I'm an Architect: Classic MLOps was built for models where you train once, deploy, and retrain occasionally. LLMs flip this. The model (GPT-4o, Claude) is a pre-trained product you license — you almost never retrain it. What changes daily is the prompt: the instruction you send the model. A prompt is a few hundred words of text, but changing it can be as consequential as releasing new software. A prompt that said "always recommend our premium product first" and gets changed to "recommend based on customer need" is a major behavioural change — and if done without testing and versioning, it's a production incident waiting to happen. LLMOps recognises prompts as first-class production artifacts that need the same engineering discipline as code.
Key differences
Classic MLOps was designed for models with structured inputs and outputs. LLMs break several foundational assumptions:
AssumptionClassic MLLLMs
What changes between versionsModel weightsModel weights + prompts + retrieval config
How you evaluate qualityMetrics on labelled test setLLM judges, human preference, task completion
Training frequencyWeekly / monthly retrainingPre-trained base is stable; prompts change daily
What "deployment" meansModel endpoint updateModel + prompt + RAG index + tool config
Primary cost driverTraining computeInference token consumption
Primary failure modeAccuracy degradationHallucination, instruction following failure, safety violation
Rollback unitModel versionModel + prompt version (must roll back together)

The most important LLMOps shift: prompts are production artifacts, not configuration strings. A prompt change can change the model's behavior as dramatically as retraining it. Prompts need: version control, automated testing before deployment, staged rollout capability, and a rollback procedure.

⚠ Common Misconception: "Prompt changes are safe to push directly — they're not code." Wrong. A prompt change is a behaviour change. In LLMOps, a prompt is treated exactly like a code change: it goes through version control, a CI/CD pipeline that runs a golden test suite, staged rollout to a subset of traffic, and has a documented rollback path. "It's just a config change" has caused production incidents that took hours to diagnose because there was no test coverage.
🎯 Explain Like I'm an Architect — Prompt Regression: Imagine your team has a carefully tested customer service prompt that works perfectly with GPT-4. Your model provider releases GPT-4o, which is faster and cheaper. You swap it in — but GPT-4o interprets one instruction slightly differently, and 12% of responses now use a tone your brand guidelines prohibit. Your old prompt "broke" on a new model. This is prompt regression — a prompt that worked on one model version fails on a newer one. LLMOps prevents this by running your full golden test suite against any new model version before you switch, exactly as you'd run unit tests before deploying new code.
4.3 CI/CD for ML Systems
🎯 Explain Like I'm an Architect: You already understand CI/CD for software: code is committed → automated tests run → if tests pass, it deploys → if tests fail, it doesn't. ML CI/CD is the same idea but the "tests" are different. Instead of unit tests checking that a function returns the right value, ML gates check: "Does this new model version score at least as well as the current production model on our test set? Does it pass our safety evaluation? Does it behave correctly on our 200 curated golden examples?" Only if all these gates pass does the model advance to the next stage. An evaluation harness is the automated exam that runs before every deployment — you can think of it as your model's driving test before it's allowed on the road.
Four gate types
  • Gate 1 — Data quality checks: schema conformance, distribution checks, coverage checks, PII scan — before training begins
  • Gate 2 — Training validation: training loss converging, evaluation metric exceeds current production model's score, no significant regression on a benchmark slice
  • Gate 3 — Pre-deployment evaluation (LLM-specific): golden test set pass rate above threshold, LLM judge quality score above threshold, safety eval, regression check vs current version
  • Gate 4 — Staged rollout: shadow mode → canary (5–10% of live traffic) → full rollout only after canary metrics are stable
🎯 Explain Like I'm an Architect — Shadow Mode & Canary Deployments: Imagine a restaurant chain testing a new menu item. They don't replace the entire menu on day one. First, they give the new dish to kitchen staff to evaluate alongside the current dish (shadow mode — both run in parallel, but only the current dish is served to customers). Then they add the new dish to the menu at 2 branches out of 200 (canary — real customers, real feedback, limited blast radius). Only if that test goes well does it roll out to all 200 branches. In AI, shadow mode means the new model processes real traffic but its answers are not shown to users — you compare quality scores between old and new. Canary means the new model answers real users, but only 5–10% of them. If something goes wrong, 90–95% of users were unaffected and you roll back.
CI/CD YAML pattern
ML CI/CD Pipeline — Schematic
pipeline: on: [data_pipeline_complete, prompt_change_merged] stages: - name: data_validation gates: [schema_check, distribution_check, pii_scan] on_failure: block_pipeline, alert_data_team - name: training trigger: data_validation_passed artifact: model_weights_v{version} - name: evaluation gates: [test_set_accuracy, llm_judge_score, safety_eval] threshold: {accuracy: 0.87, judge_score: 4.2, safety: 0.995} on_failure: block_deployment, create_review_ticket - name: staging strategy: shadow_traffic duration: 24h gates: [quality_parity_with_prod] - name: production strategy: canary_5pct → canary_25pct → full rollback_on: {error_rate: 0.05, quality_score_drop: 0.1}
4.4 Drift Detection — A Unified Framework
🎯 Explain Like I'm an Architect — Data Drift: Your fraud detection model was trained on 2023 spending data. It learned that "purchases over £500 at electronics stores on weekday evenings" is a fraud signal. Then March 2024 arrives, there's a big promotion, and legitimate customers start buying expensive electronics in exactly that pattern. The model starts flagging real customers as fraudsters. Your data hasn't changed — your code hasn't changed — but the world has changed. The inputs arriving at the model now look different from the inputs it was trained on. That's data drift: the input population has shifted away from the training population. Think of it this way: you trained on summer customers, but you're serving winter customers. The model doesn't know the season changed.
Type 1 — Data Drift
Input distribution shift: the data coming into the model at inference time looks different from the training data.
Example: a product recommendation model trained on summer purchase data is now seeing winter purchase patterns.
Detection: compare statistical distribution of incoming features against training distribution. Tools: PSI (score above 0.2 = significant shift), KL divergence, KS test, chi-squared.
Alert when: PSI > 0.2 on any key feature, or incoming distribution is more than 2 standard deviations from the training mean.
🎯 Explain Like I'm an Architect — PSI, KL Divergence, and KS Tests (without the statistics): These are all alarms that tell you "the population of inputs the model is seeing has changed significantly from the population it was trained on." Think of them as sensors. PSI (Population Stability Index) is like a thermometer — a score above 0.2 is the equivalent of the thermometer reading above 38°C: something is wrong, investigate. KL divergence and KS tests are different sensors measuring the same thing in different ways. You don't need to understand the maths — you need to configure the threshold (PSI > 0.2 triggers an alert) and have a process for what happens when the alarm sounds. An architect's job is to ensure these sensors exist and are connected to the alerting system.
Type 2 — Concept Drift
Label distribution shift: the relationship between inputs and correct outputs has changed, even if inputs look the same.
Example: a fraud detection model trained in 2024 encounters a new fraud pattern that looks similar to legitimate transactions.
Detection: track model accuracy on a labelled holdout refreshed with recent data. If accuracy drops without input distribution changing, it's likely concept drift.
Alert when: accuracy on the rolling 30-day labelled holdout drops more than X% from the deployment baseline.
⚠ Common Misconception: "Data drift and model drift (concept drift) are the same thing." They're not — and the difference matters for what you do about it. Data drift = the inputs changed (the world changed around the model). Concept drift = the correct answers changed (the relationship between inputs and outputs changed). With data drift, you may need to retrain on new data. With concept drift, you definitely need new labelled examples showing the new pattern. An example: a spam filter can experience both simultaneously — new spam types arrive (concept drift) and legitimate email patterns also change over time (data drift).
Type 3 — LLM Output Drift
Quality / behavior shift: the model's outputs change in quality or character without any explicit change to the system.
Causes: the model provider silently updates the model; the retrieval index content has changed; the distribution of incoming queries has shifted.
Detection: LLM judge scores on a golden test set run on a schedule. Compare score distribution week-over-week.
Alert when: mean LLM judge score drops more than 0.3 points below the deployment baseline.
Type 4 — Prompt Drift
System prompt assumptions no longer match reality. Detection: golden test set regression. (Covered in detail in the Agent Runtime Ops module.)

4.5 LLM Evaluation — The New Discipline
🎯 Explain Like I'm an Architect — What an Evaluation Harness Is: For traditional software, an automated test suite checks "did this function return the right answer?" For an LLM, there is often no single right answer — a customer service response can be good in many ways. An evaluation harness is the automated exam that runs before every model or prompt deployment. It has three parts: (1) a golden test set — 200+ curated examples of inputs with known-good expected outputs; (2) an LLM judge — a capable AI that reads the model's actual response and scores it against dimensions like accuracy, tone, safety; (3) pass/fail thresholds — if the average score drops below 4.1 out of 5, the deployment is blocked. Think of it as your quality assurance process, but automated and running in CI/CD. No model or prompt ships without passing this exam.
Three evaluation layers
  • Layer 1 — Automated metrics (fast, cheap, imperfect): BLEU/ROUGE (text overlap — cheap but misses semantic correctness), exact match (works when one correct answer exists), perplexity (lower = better; useful for comparing versions on same domain)
  • Layer 2 — LLM judge (better quality, moderate cost): a capable LLM (e.g., GPT-4o or Claude Opus) evaluates production model outputs against a rubric. Dimensions scored separately: accuracy, completeness, groundedness, safety, tone and format.
  • Layer 3 — Human evaluation (gold standard, expensive): human annotators evaluate a sample of outputs. Costs 50–200× more than LLM judging. Use it to calibrate your LLM judge, not as primary monitoring.
The eval loop
2026 LLMOps Eval Loop
Production traffic │ ▼ Automatic sampling (e.g., 5% of requests) │ ▼ LLM judge scoring (near real-time) │ ├── Score above threshold → log, no action ├── Score below threshold → flag for human review queue └── Score trend declining → drift alert → pipeline trigger │ ▼ Human review queue (for flagged samples) │ ▼ Reviewed samples → fine-tuning dataset / golden test set updates │ ▼ Updated golden test set → next CI/CD pipeline run

This is the production traffic feeds the eval set; the eval set gates every release loop. Better monitoring produces better training data, which produces better models, which produce more reliable monitoring signals.

4.6 The Feature Store
🎯 Explain Like I'm an Architect — What a Feature Store Is: Imagine four different teams all need to know "how much has this customer spent in the last 30 days?" Without a feature store, each team writes their own version of that calculation — slightly different logic, different data sources, different update schedules. The recommendation model, the fraud model, and the churn model all compute "30-day spend" differently. When things go wrong, nobody can compare results. A feature store is the shared pantry of pre-computed ingredients. One team defines "30-day spend" once, computes it in one place, and every model that needs it takes it from the same shelf. Everyone is cooking with the same ingredients. The additional benefit: what the model trains on (historical 30-day spend) and what it gets at inference time (today's 30-day spend) come from the same computation logic — eliminating a major class of silent failures.
4.7 Experiment Tracking
🎯 Explain Like I'm an Architect — Why Experiment Tracking Matters: A data scientist runs 50 training experiments over two weeks, tweaking hyperparameters and data preprocessing each time. Without tracking, by experiment 50, nobody can remember which combination of settings produced the best result — or reproduce experiment 23 that looked promising. Experiment tracking is a lab notebook for every training run. Every experiment automatically records: what data it used (version), what parameters it trained with, what metrics it achieved, and which model artifact it produced. When you want to understand why this week's model is worse than last week's, you can compare the two experiment records side by side — data version, code version, parameters, and results — and pinpoint the change that caused the regression.
4.8 Observability — OpenTelemetry for LLMs

Architecture Perspective
The MLOps Maturity Model
Level 1 — Manual MLOps
  • Training: by hand in notebooks
  • Deployment: manual, no CI/CD
  • Monitoring: someone checks a dashboard occasionally
  • Rollback: "redeploy the old version manually"
  • Suitable for: proof-of-concept, single model, low traffic
Level 2 — Automated Pipeline
  • Training: automated pipeline triggered by data or schedule
  • Deployment: CI/CD with automated eval gates
  • Registry: versioned model artifacts
  • Monitoring: drift alerts with defined thresholds
  • Suitable for: production systems, multiple models, regulated use cases
Level 3 — Continuous Learning
  • Eval: production traffic automatically samples into eval set
  • Gates: eval set gates every pipeline run
  • Retraining: triggered automatically by drift signals
  • Deployment: shadow/canary standard for every release
  • Governance: full cost governance with per-team chargeback
  • Suitable for: business-critical AI, multiple teams, enterprise scale
Most enterprise AI initiatives in 2026 are at Level 1 or early Level 2. The gap between Level 1 and Level 2 is where most production incidents happen.

Check Yourself
Test your understanding — 5 questions
  1. Six months after deploying a product recommendation model, accuracy dropped from 87% to 71% with no code changes. Your colleague says "the model must be broken." What are the two most likely real causes, and how would you investigate each?
    Answer: The two most likely causes are data drift (the input population has changed — seasonal shift, new product catalogue, changed user demographics) and concept drift (the relationship between inputs and correct outputs has changed — what users want from a recommendation has shifted). To investigate data drift: compare the statistical distribution of incoming features today against the training data baseline using PSI scores per feature — any feature with PSI > 0.2 is a candidate. To investigate concept drift: measure accuracy on a labelled holdout set refreshed with recent user interactions — a drop without input distribution change points to concept drift.
    Simple explanation: The model didn't break — the world changed and the model didn't know about it.
    Architecture takeaway: Every production AI system needs drift monitoring from day one. When designing the system, budget for (1) a scheduled data distribution comparison job, (2) a rolling labelled holdout dataset refreshed with recent data, and (3) a defined threshold that triggers a retraining pipeline automatically. Without these, accuracy degradation is silent until users complain.
  2. Your team wants to update the system prompt for a customer service LLM. A developer says "it's just a config change, I'll push it directly to production." What's the correct MLOps response, and what specific risks does skipping the process create?
    Answer: The correct response is: no prompt change goes directly to production. The prompt change must go through the same pipeline as a code change — commit to the prompt registry, trigger the CI/CD evaluation pipeline, run the full golden test set (200+ curated Q&A pairs) through the LLM judge with the new prompt, verify pass rate above threshold, deploy to shadow/canary first, monitor quality metrics before full rollout. The specific risks of skipping: (1) behaviour regression — the new prompt may subtly change responses in ways that violate brand guidelines, legal requirements, or user expectations that your test suite would have caught; (2) no rollback artifact — if you pushed directly, rolling back requires manually reconstructing the previous prompt from memory or git history; (3) no audit trail — for regulated industries, you cannot demonstrate what changed and when if an adverse customer outcome is investigated.
    Simple explanation: A prompt change is a behaviour change — treat it like a code release, not a spelling correction.
    Architecture takeaway: Store all prompts in a prompt registry (a versioned repository — could be as simple as a git folder with a naming convention). Wire the prompt registry to your CI/CD pipeline so any merge to the production branch triggers the evaluation harness automatically. Make "prompt deployed without passing eval" impossible by design.
  3. What is an LLM judge and why is it preferable to BLEU/ROUGE for evaluating a customer service LLM? What are its limitations?
    Answer: An LLM judge is a capable large language model (e.g., GPT-4o, Claude Opus) configured to evaluate another model's outputs against a scoring rubric. You give it: (a) the original user query, (b) the model's response, and optionally (c) a reference answer — and ask it to score dimensions like accuracy, completeness, groundedness, safety, and tone on a 1–5 scale. BLEU and ROUGE measure text overlap — they check whether the model's response contains the same words as a reference answer. For customer service, two responses can be equally correct while being phrased completely differently. BLEU/ROUGE would score the phrased-differently response poorly even if it's semantically better. An LLM judge understands meaning, not just word matches. Limitations: LLM judges have their own biases (they tend to prefer longer, more confident-sounding responses), they add inference cost, and they can be inconsistent — requiring calibration against human annotators. LLM judge scores should be calibrated quarterly against a human-labelled sample to ensure the judge's standards haven't drifted.
    Simple explanation: BLEU/ROUGE checks whether the words match; an LLM judge checks whether the answer is actually good.
    Architecture takeaway: Design your evaluation harness with a tiered approach: BLEU/ROUGE for fast regression detection (cheap, runs on every commit), LLM judge for quality scoring (moderate cost, gates every deployment), human review for quarterly calibration (expensive, small sample). Never use BLEU/ROUGE alone as your quality gate for free-text AI systems.
  4. Describe the "production traffic feeds the eval set; the eval set gates every release" loop. Why does an AI system that closes this loop continuously improve, while one that doesn't gradually degrades?
    Answer: The loop works like this: (1) production traffic is continuously sampled (e.g., 5% of requests); (2) an LLM judge scores each sampled response automatically; (3) responses that score below threshold are flagged for human review; (4) human reviewers classify them — correct/incorrect, what the right answer should have been; (5) the reviewed examples are added to the golden test set; (6) the expanded golden test set now gates every future CI/CD pipeline run. A system that closes this loop improves because: its test coverage expands to include real failure patterns from production (not just patterns the team could imagine at launch); each production failure that makes it into the test set prevents that failure from passing QA undetected in future. A system that doesn't close this loop degrades because: the world changes (new question types, new user needs, new edge cases emerge), but the test set stays static — the CI/CD gates become less representative of real production quality over time, and the model can "pass" the test suite while failing users.
    Simple explanation: Every time the system fails in production and you capture that failure in your test set, you make the system smarter about not failing that way again.
    Architecture takeaway: Design the feedback loop into the system from day one. The pipeline needs: a sampling mechanism on the production request handler, a flagging queue fed by the LLM judge, a human review workflow (JIRA tickets or a custom tool), and a defined process for promoting reviewed examples to the golden test set. The review workflow is often the bottleneck — design for it, don't assume it will happen organically.
  5. Your finance director asks why the AI knowledge assistant cost 40% more than budgeted last quarter. What monitoring would have caught this earlier, and what are the two most likely root causes to investigate?
    Answer: Monitoring that would have caught this: per-team, per-query-type token cost tracking with weekly budget alerts set at 110% of baseline. Specifically: (1) track average input tokens per request and output tokens per request as daily metrics — a sustained upward trend is an early warning; (2) set a budget alert that fires when weekly spend exceeds the baseline by >15%; (3) track cost per query by team and query category, not just total cost. Two most likely root causes: (a) Context bloat — the RAG retrieval is returning more chunks per query than necessary (e.g., top-10 instead of top-3), adding thousands of tokens per request. Fix: reduce retrieved chunk count, test that quality doesn't drop; (b) Query volume increase in a high-cost category — certain query types (e.g., complex multi-document financial analysis) consume 3–5× more tokens than average. If one team's query volume grew unexpectedly, total cost grows disproportionately. Fix: check per-team query volume growth against the budget model.
    Simple explanation: LLM costs grow with every token processed — if nobody is watching the per-query token count trend, cost overruns arrive as surprises.
    Architecture takeaway: Treat LLM token cost as an operational metric, not just a finance metric. Wire cost-per-request alerts into the same observability platform as latency and error rate alerts. Set budget envelopes per team with automated alerts, and include cost-per-query in your weekly operational review dashboard. A 40% cost overrun discovered quarterly is a governance failure — it should have been caught in week 2.

8.1 Feature Stores and Why They Matter

A feature store is a centralized platform for storing, sharing, and serving ML features — the engineered inputs to your models. Without one, near-duplicate features proliferate across teams, computed slightly differently, each maintained separately.

  • Centralised feature definitions: one canonical definition of "customer 30-day purchase value"
  • Consistent train/serve: the same feature computation logic runs at training time and inference time — eliminating the most common source of train-serve skew
  • Point-in-time correctness: for training, the feature store can reconstruct what the feature value was at any historical timestamp — critical for avoiding data leakage in time-series models
  • Offline store (batch): historical feature values in a data warehouse, used for training
  • Online store (real-time): current feature values served at inference time with low latency (typically Redis or DynamoDB backed)
8.2 Train-Serve Skew

Train-serve skew is the silent killer of ML systems: the model was trained with features computed one way, but at inference time the same features are computed slightly differently.

Example: during training, "days since last purchase" used floor(). At inference time, the serving pipeline uses ceil(). Off by at most 1 day — but enough to shift the score distribution and degrade precision.

This is why the feature store's train/serve consistency guarantee matters. Model performance in shadow mode often differs from test set performance: if the online feature computation differs from the offline version, the model operates on data it was never trained on.

Detection: compare the statistical distribution of features in training data vs. features in the online feature store. Any systematic difference is a skew candidate.

8.3 Reasoning Model Cost Management

Models with extended thinking (o3, DeepSeek R1, Claude Opus with extended thinking) charge for reasoning tokens — often at 2–4× the rate of standard output tokens.

  • Budget tokens per query tier: simple factual queries get a small thinking budget (e.g., 1K tokens); complex multi-step reasoning gets a larger budget (e.g., 16K tokens). Route by query complexity classifier.
  • Thinking token monitoring: track reasoning.budget_utilization per query type. If a query type consistently uses 100% of its budget, consider increasing it — hitting the budget cap degrades answer quality.
  • Cached reasoning: for common reasoning patterns, some providers support prompt caching that includes the reasoning trace — you pay once for the reasoning, then reuse the cached result for identical queries.

Key Takeaways
  • MLOps is the discipline that bridges AI experiments and production reliability. Without it, you have no reproducibility, no automated quality gates, no rollback capability, and no early warning when models degrade. The gap between Level 1 (manual) and Level 2 (automated pipeline) is where most production AI incidents originate.
  • Prompts are production artifacts — treat them like code. A prompt change can alter model behavior as dramatically as retraining. Version control, automated testing, staged deployment, and rollback procedures apply to prompts exactly as they apply to code. "It's just a config change" is the most dangerous phrase in LLMOps.
  • Drift comes in four types, each needing different detection. Data drift (inputs changed), concept drift (correct outputs changed), LLM output drift (quality degraded), and prompt drift (context assumptions changed). A monitoring strategy that only watches error rates will miss all four — they typically manifest as silent quality degradation, not system errors.
  • The eval loop is the core of LLMOps maturity. Production traffic samples → LLM judge scoring → human review of failures → golden test set updates → gates on every release. The teams that close this loop continuously improve their systems. The teams that don't are operating blind.
  • Cost is a first-class operational signal, not an afterthought. LLM costs are variable, runtime-determined, and can spike 10–100× due to context bloat, retry loops, or unexpected volume. Per-team, per-model cost tracking with weekly budget alerts is as important as latency monitoring — and usually catches system anomalies faster.
