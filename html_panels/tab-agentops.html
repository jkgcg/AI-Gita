What Is Agent Runtime Ops?
You have deployed an AI agent. It is handling customer queries, processing returns, routing tickets. It worked perfectly in testing. Agent Runtime Ops is everything you do to keep it working in production — and to know when it stops.

This is the operational layer that sits on top of the agent platform. If the Agentic Platform module covered how to build an agent system, Agent Runtime Ops covers how to run it: memory management, monitoring, human-in-the-loop gates, prompt drift detection, and legacy system connectors.

Key distinction from traditional app operations: in a conventional system, you monitor errors. In an agent system, the scariest failures produce no errors — the agent confidently does the wrong thing, logs a success, and moves on.

Why Should I Care
For Solution Architects

Every operational decision you know — SLA, rollback strategy, incident runbook, capacity planning — has an agent-specific equivalent that behaves differently:

  • SLA: a traditional API call has deterministic latency. An agent call has variable latency — 3 tool calls takes 5 seconds; an unexpected loop takes 45 seconds. Your SLA must cover the distribution, not just the average.
  • Rollback: rolling back an agent version does not undo the actions it already took. Blast radius controls are your primary defence, not rollback.
  • Capacity: agents make multiple LLM calls per task. A sudden spike in agent activity translates directly to API cost — not just server load. Cost alerting is as critical as latency alerting.
For Enterprise Architects

The operational maturity model for agent systems is 2–3 years behind traditional application operations. Most teams are figuring out runbooks, dashboards, and escalation paths as they go.

Plan for observability investment upfront, not as a retrofit. The hardest thing to add after deployment is a comprehensive trace of why the agent made every decision it made. OpenTelemetry instrumentation, prompt logging, tool call capture, and the reasoning chain must be built in from day one.

Think About It Like This — Air Traffic Control
Imagine you are running an air traffic control operation. You have many aircraft (agents) in the air simultaneously, each making decisions based on instructions and sensor readings.
  • Memory management is like each aircraft's flight computer — limited memory, keeps the last N radar contacts, discards older ones. Before discarding, it writes key facts to the flight recorder.
  • Monitoring is the radar display — you are not watching every aircraft continuously. You watch for signals that indicate something needs attention: unexpected trajectory changes, radio silence, entering a no-fly zone, fuel below threshold.
  • HITL gates are the no-fly zones and mandatory clearances. Certain actions require tower clearance regardless of how confident the aircraft's computer is. The aircraft cannot proceed autonomously past those gates.
  • Prompt drift is like navigational chart updates. If the airspace boundaries changed last month and your charts were not updated, the aircraft follows correct procedures for the wrong map. Everything looks fine on the instruments, but you are heading somewhere unintended.
Key insight: air traffic controllers do not control the aircraft directly — they control the boundaries and escalation triggers. That is exactly what Agent Runtime Ops does.
Step-by-Step — The Core Concepts
4.1 Four-Layer Memory Architecture
Using the context window for everything is expensive, slow, and does not persist across sessions. The production pattern is four distinct memory layers with the right storage type, retrieval pattern, and cost profile for each purpose.
Layer 1 — Working Memory (In-Context)
What the agent holds in the LLM context window right now: current conversation, last few tool call results, task instructions. Capacity: 8K–32K tokens reserved. Lifespan: one task or session. Cost: high — every token is paid on every LLM call. Use for: active conversation, current task state, immediate tool results.
Layer 2 — Episodic Memory (Short-Term Persistent)
A log of what happened in recent sessions: past conversations, prior actions, outcomes observed. Storage: relational or document store, not the LLM context. Retrieval: query for relevant past episodes before starting a new task. Lifespan: days to weeks, then summarised or archived. Use for: cross-session continuity, personalisation, avoiding repeating questions the customer already answered.
Layer 3 — Semantic Memory (Knowledge Base)
Structured knowledge the agent can retrieve: product catalogues, policy documents, FAQs, process guides. Storage: vector database (semantic search) + document store (full retrieval). Retrieval: RAG — embed the query, find relevant chunks, inject into working memory. Use for: policy lookups, product information, process guidance — anything that would otherwise bloat the context.
Layer 4 — Procedural Memory (Agent Configuration)
The agent's "how to behave" layer: system prompt, persona definition, tool permissions, escalation rules. Storage: configuration files / agent registry. Lifespan: versioned and deployed, not ephemeral. Use for: role definition, tone, boundaries, escalation thresholds, tool access scope.

Context budget rule — typical production allocation for a 128K window:

128K context window ├── 2K — System prompt (procedural memory) ├── 20K — Retrieved knowledge (semantic memory, per-request) ├── 10K — Episodic context (relevant past interactions) ├── 40K — Working memory (current conversation + tool results) └── 56K — Reserved for model reasoning + output generation Exceeding the budget forces truncation. Design the truncation policy before it happens in production.

4.3 HITL Gate Design — Risk-Tiered Approvals
Human-in-the-Loop (HITL) gates are points where the agent must pause and wait for human approval. Getting this wrong in either direction is costly: too few gates and the agent makes consequential mistakes autonomously; too many gates and you have automated nothing — just added an LLM wrapper to manual work. Design principle: tier approvals by risk, not by task type.
Risk LevelCriteriaGate Behaviour
LowRead-only, reversible, low-value, customer-initiatedNo gate — agent proceeds autonomously
MediumWrites to a system, value below threshold, standard caseSoft gate — agent proceeds, logs for async human review
HighIrreversible action, value above threshold, or non-standard caseHard gate — agent pauses, notifies human, waits up to SLA
CriticalPII deletion, regulatory-impacted, security-sensitiveMandatory human approval — agent cannot proceed without explicit sign-off

4.4 Prompt Drift Detection
Your agent prompts encode assumptions about: the tools that exist, the data formats tools return, the policies the agent should follow, and the model it is running against. Any of these can silently degrade agent performance without a single error being thrown.
Drift Detection Pipeline
  1. Golden test set: 50–200 curated test tasks with known correct outputs — covering edge cases, standard cases, and past production failures
  2. Scheduled regression: run the golden test set daily or on every deployment through the live agent
  3. Scoring: an LLM judge evaluates each output against the reference answer — scores correctness, completeness, and policy compliance
  4. Drift alerting: if mean score drops more than X% from baseline, trigger an alert and flag for investigation
Retrieval Quality Signals
  • Retrieval precision drop: knowledge base documents have drifted from query patterns
  • Low-similarity retrievals: queries are no longer finding semantically relevant content
  • Stale document rate: retrieved documents have not been updated in more than N days
  • Coverage gap: query classes that return no results above threshold

Principle: treat prompt and retrieval quality as first-class operational signals, not one-time configuration concerns.

4.5 Enterprise Legacy Connectors
Most enterprises have critical data in systems that predate REST APIs: SAP ERP, IBM MQ, TIBCO, COBOL mainframes. Design principle: do not expose legacy systems directly to agents. Wrap them in the integration middleware layer you already operate.
Pattern A — REST Facade

Recommended for new work. Build a REST/OpenAPI endpoint in your integration middleware (MuleSoft, Azure APIM, IBM APIC) that wraps the legacy system. Register this endpoint as a tool in the tool registry.

Agent → Tool Registry → REST Facade (MuleSoft) → SAP RFC → SAP ERP
Pattern B — Event-Driven

For real-time data. Connect the agent's knowledge plane to the event stream. The agent's semantic memory is updated in near-real-time from the event feed rather than via synchronous API calls.

SAP ERP → IBM MQ → Event Processor → Agent Knowledge Plane Agent reads from knowledge plane (no synchronous SAP call at inference)
Pattern C — Async Task Dispatch

For long-running processes. The agent dispatches a task message to the queue and registers a callback. The agent suspends; the runtime resumes it when the callback arrives.

Agent → dispatch_warehouse_task() → Message queued to WMS → Agent suspends, state persisted ← WMS completes, sends event → Runtime resumes agent with result
Enterprise Example — Supply Chain Exception Handler
Scenario: 2,000 shipments tracked daily, 150–200 exceptions per day

Memory architecture for this agent:

  • Working memory: current exception case — shipment ID, delay reason, carrier status, impacted orders
  • Episodic memory: previous exceptions on the same carrier route (is this carrier frequently delayed in December?)
  • Semantic memory: supplier SLA agreements, escalation playbooks, carrier contact details, customs documentation requirements
  • Procedural memory: system prompt defining escalation rules, tool permissions, communication tone

Monitoring dashboard:

Investigation finding: the carrier API introduced a new rate limit; the agent retries failed calls, each retry costs LLM tokens to process the error, and 3 retries per failed call × 6% error rate is compounding cost. Fix: add a circuit breaker — after 2 failed calls in a row, fail fast and escalate rather than retrying.

Prompt drift example: in March, the supplier SLA was updated — penalty clauses now kick in at 3-day delays instead of 5. The semantic memory (SLA document) was updated but the procedural memory (system prompt) still said "escalate if delay > 5 days." The drift went undetected for 2 weeks because no errors were thrown. The golden test set caught it: a test case with a 4-day delay was not escalating. Fix: update the system prompt and add the 4-day delay scenario to the golden test set.

Agent Version Management
Agent Version = 4 Components

An agent's behaviour is a combination of all four — and all four can change independently:

{ model: "gpt-4o-2025-01-15", prompt: "customer-service-v2.3", tools: "tool-registry-snapshot-2026-06-15", knowledge: "knowledge-base-2026-06-20" }
Shadow Mode Deployment

A/B testing requires routing at the task level, not the request level — you do not want a multi-turn conversation to switch agent versions mid-session.

Shadow mode: the new agent processes the same tasks as the live agent but its outputs go to a review queue rather than the customer. Compare new vs. old outputs using an LLM judge before promoting the new version to live traffic.

Check Yourself — 5 Questions
Self-Assessment

Use these to verify you can reason through operational architecture — not just recall definitions.

  1. An agent resolving 90% of refund requests autonomously last month now resolves only 70%, with no errors in the logs. What are the first three things you investigate? — (1) Prompt drift: did the return policy change? Run the golden test set to check benchmark scores. (2) Retrieval quality: are knowledge base documents still accurate and fresh? Check retrieval precision and document freshness. (3) Model change: did the model gateway route to a different model version? Check model version in LLM call logs around the time the metric changed.
  2. Your team wants to deploy an agent that sends promotional emails to customer segments based on purchase history. What HITL gate design would you propose? — Sending bulk communications is irreversible and high blast-radius. Critical-tier HITL gate: the agent prepares the draft, selects the segment, and presents both to a human approver before any send. Approval includes: segment size, sample recipients, email preview, estimated unsubscribe risk.
  3. What is the difference between episodic memory and semantic memory? Give a concrete supply chain example of each. — Episodic memory is a record of what actually happened: "Carrier X was 3 days late on the Milan route in November and again in December — flagged as a seasonal pattern." Semantic memory is structured knowledge independent of experience: "The SLA with Supplier Y states delays exceeding 3 days trigger a 2% penalty clause." Episodic is retrieved by temporal/entity-based lookup; semantic by semantic similarity.
  4. Why is cost monitoring as critical as latency monitoring for agent systems? — In conventional services, cost is tied to infrastructure (fixed) — load spikes hit latency before cost. In agent systems, cost is proportional to LLM token consumption, which is runtime-determined by agent behaviour. A looping agent or one retrying errors without a circuit breaker can consume 10–100× expected tokens with no latency alert triggering. Cost alerting with per-agent and per-task caps is a hard requirement, not a nice-to-have.
  5. Your operations team asks for a runbook for "agent is producing wrong answers." What are the five diagnostic steps? — (1) Check the OTel reasoning trace: what tools did the agent call, what did they return, and what was the LLM reasoning at the decision point? (2) Check retrieval quality: were retrieved documents current and relevant? (3) Check model version: did the gateway change routing recently? (4) Run the golden test set: is the benchmark score below baseline, confirming systematic degradation? (5) Check tool results: did tools return correct data? If tools returned stale or wrong data, the problem is data quality, not agent quality.

Agent Task Tree Visualization

Complex multi-agent workflows create a task tree: a hierarchical structure of parent tasks spawning child tasks, with dependencies between them. Visualising the task tree in your monitoring dashboard shows immediately which branches are stuck, which tools are failing, and where HITL gates are creating queues.

Task: "Resolve supply chain exception batch (50 exceptions)" ├── Sub-task: "Resolve EX-001" [autonomous] │ ├── Tool: get_carrier_status → delayed │ ├── Tool: get_impacted_orders → 3 orders │ └── Tool: notify_operations → sent ├── Sub-task: "Resolve EX-002" [HITL gate triggered] │ ├── Tool: get_carrier_status → critical delay │ ├── HITL: escalation created → waiting for supervisor │ └── [suspended] └── Sub-task: "Resolve EX-003" [failed] ├── Tool: get_carrier_status → API error (timeout) ├── Tool: get_carrier_status → API error (timeout) [retry 1] └── Circuit breaker triggered → escalated to human
Context Window Pressure Strategies

When working memory approaches the context budget:

  • Summarization: call the LLM to compress conversation history into a compact summary. Cost: one extra LLM call. Gain: 70–80% context reduction. Risk: lossy — some details in raw history may not survive.
  • Structured state extraction: extract key facts into a structured JSON object at each step. A JSON state object is ~200 tokens; the raw conversation that produced it is ~2,000 tokens. 10× compression with zero information loss for the decision the agent needs to make.
  • Memory offload: move tool results to episodic memory after each step. The agent retains a summary ("carrier API confirmed 2-day delay"), not the full raw response (which might be 1,500 tokens of JSON).
// Structured state — ~200 tokens vs ~2,000 tokens of raw conversation { "customer_id": "C-12345", "issue_type": "missing_item", "order_id": "O-67890", "items_ordered": 3, "items_received": 2, "missing_item": "SKU-ABC", "warehouse_scan_weight_discrepancy": true, "refund_eligible": true, "refund_amount": 79.99 }
Circuit Breakers for Tool Calls

Borrowed from distributed systems, a circuit breaker wraps tool calls to prevent a failing external system from cascading into agent failure. For agents it serves a second purpose: preventing retry loops from consuming tokens.

  • Closed (normal): tool calls proceed normally
  • Open (failing): after N consecutive failures, the circuit opens — calls fail immediately without hitting the external system
  • Half-open (recovering): after a timeout, one probe call is allowed through; if it succeeds, the circuit closes again
Tool call: carrier_api.get_status() ↓ Circuit breaker check: - Error count in last 60s: 4 - Threshold: 3 - State: OPEN ↓ Fast fail → return: {status: "circuit_open", escalate: true} ↓ Agent: "Tool unavailable — escalating to human"
Key Takeaways
  • Agent failures are often silent. The agent confidently does the wrong thing, logs a success, and moves on — no errors thrown. Your monitoring strategy must watch for behavioral anomalies (unexpected tool sequences, cost spikes, quality score drops), not just error rates.
  • Design HITL gates by risk tier, not by task type. The question is not "which tasks need human approval?" but "which outcomes are irreversible, high-value, or regulatory-impacted?" Low-risk reversible actions should be autonomous. Irreversible or high-value actions need hard gates — regardless of how confident the agent is.
  • Four-layer memory is not optional overhead — it is cost and quality management. Putting everything in the context window is expensive and unreliable at scale. Working memory for the current task, episodic memory for history, semantic memory for knowledge, and procedural memory for behavior — each layer has the right storage type, retrieval pattern, and cost profile for its purpose.
  • Prompt drift is a production reliability concern, not a one-time setup task. Your system prompt, knowledge base, tool definitions, and model version all change over time. A golden test set run on a schedule is the only reliable way to detect when those changes have silently degraded agent quality.
  • The observability stack for agents requires LLM-specific extensions. Standard application traces (latency, errors, throughput) are necessary but not sufficient. You need reasoning traces, tool call graphs, and LLM quality signals. Instrument with OTel + GenAI semantic conventions from day one — not as a retrofit after an incident.
