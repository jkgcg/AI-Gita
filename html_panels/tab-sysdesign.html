What Is It (Plain English)
AI system design is the discipline of architecting production AI systems — deciding how all the pieces fit together so the system is reliable, cost-effective, scalable, and governable.
It's not the same as designing a traditional distributed system. Not because the principles are different, but because AI systems have properties that traditional systems don't:
  • Non-deterministic outputs: the same request can return different responses. You can't cache them the same way. You can't unit-test them the same way. SLAs look different.
  • Variable compute cost: processing a 50-token query is 20× cheaper than a 1,000-token query. Cost is runtime-determined, not fixed per request.
  • Quality as a dimension: a web API either returns 200 or it doesn't. An LLM returns 200 whether the answer is correct or hallucinated. Quality is a first-class design concern.
  • Model as a dependency: your system depends on a model that can change, degrade, or become unavailable. The model is an external dependency with its own versioning, pricing, and SLA — like a database, but less predictable.
  • Context as state: LLMs are stateless by default, but the quality of their response depends entirely on what's in the context window. Managing context — what goes in, in what order, how much — is a core design concern.
AI system design is the practice of making good decisions across all of these dimensions before a line of code is written. The decisions you make at design time — model choice, retrieval strategy, routing logic, caching approach, evaluation architecture — are expensive to reverse once the system is in production.
Why Should I Care
For Solution Architects

You are the person who gets asked: "Can we build this?" before anyone knows how. AI system design is how you answer that question with confidence — not by guessing, but by decomposing the problem into solvable pieces and identifying where the risks live.

  • Latency: the user expects a response in 2 seconds. A single GPT-4o call takes 1.5–4 seconds. How do you build a system that meets a 2-second P95 SLA?
  • Cost: 50,000 users × 10 queries/day × £0.03/query = £15,000/day. Is that in budget? What does the cost model look like as you scale?
  • Quality: how do you know if the system is working? What does "working" mean for an LLM that gives fluent wrong answers?
  • Failure modes: what happens when the LLM API is down? When the vector store returns no results? When the agent loops? What are the fallbacks?
For Enterprise Architects

AI systems touch more of your enterprise stack than any other category of application — they read from your data platform, call your APIs, write to your CRM, and communicate with your customers. Three design decisions with enterprise-wide implications:

  • Multi-tenancy model: how do you serve multiple teams or customers from one AI system without data bleed, cost confusion, or one tenant's load degrading another's experience?
  • Model routing strategy: which queries go to which model? A routing layer that sends complex queries to GPT-4o and simple queries to Claude Haiku can cut costs by 60–80% with no user-facing quality difference. But it needs to be designed in, not bolted on.
  • Feedback loop architecture: how does what the system does in production inform how it improves? A system without a feedback loop is frozen at the quality it had at launch. This is a design concern, not an operational one.
Think About It Like This
The Contact Centre Architecture Analogy

Imagine you're designing a contact centre for a large retailer — not building it, but designing the architecture before anyone is hired or any software is procured.

  • Routing: does every call go to the same pool of agents, or do you route by query type? Returns to the returns team, VIP customers to dedicated agents? — Model routing in AI systems
  • Tiers: do you have a self-service IVR for simple questions, a first-line agent for standard queries, and escalation to a senior specialist for complex issues? — Model tiering: fast/cheap for simple, powerful/expensive for complex
  • Context passing: when a customer is transferred from IVR to agent to specialist, what information travels with them? Do they have to re-explain their issue each time? — Context management and session memory
  • Capacity and overflow: what happens at peak? Are there overflow rules? What's the maximum queue depth before "try again later"? — Rate limiting, fallback routing, load shedding
  • Quality assurance: how do you know if agents are giving good answers? You sample, score, and use the results to train. — LLM evaluation and monitoring
  • Cost model: IVR costs £0.02/minute. First-line agents £0.50/minute. Senior specialists £2.00/minute. The architecture that routes correctly saves enormous money at scale. — LLM cost routing
Every one of these contact centre decisions has a direct AI system design parallel. The decisions you make before the system is built determine whether it's operationally manageable and financially viable at scale.

Design consideration: the routing classifier itself must be cheap and fast (< 50ms, < £0.001/call). A complex router that's almost as expensive as just using the premium model defeats the purpose.

4.4 Multi-Tenancy in LLM Systems
Four multi-tenancy concerns
  • Data isolation: Tenant A must not see Tenant B's data in retrieval, context, or logs. Options: index-per-tenant (strong isolation, higher cost) vs metadata filtering (cheaper, but a missing filter = data leak) vs hybrid (shared index for public knowledge, per-tenant for sensitive data).
  • Cost isolation: tag every LLM call with tenant ID at the model gateway layer; track token usage per tenant, per model, per time period; set per-tenant budget caps and alert thresholds.
  • Rate limiting: per-tenant rate limits at the client layer (requests/minute, tokens/minute); token bucket algorithm for smoothing burst patterns; priority queuing for premium tenants.
  • Configuration isolation: different tenants may need different system prompts, different model tiers, different tool access. Tenant configuration stored in a registry, loaded at request time. Tool permissions defined per-tenant role, not globally.
4.5 Caching Strategy
Three caching levels
  • Level 1 — Exact match (response cache): cache the full LLM response for an exact input string. Hit rate: low. Best for FAQs, common greetings, boilerplate. Storage: Redis or Memcached.
  • Level 2 — Semantic similarity (semantic cache): cache responses for semantically equivalent inputs. "What is your return policy?" and "How do I return a product?" → same cache entry. Hit rate: 5–15% in practice. Implementation: embed the query, check vector similarity against cached queries (threshold ~0.95).
  • Level 3 — Prompt caching (provider-side): OpenAI, Anthropic, and Google cache the KV computation for the static portion of your prompt. If the first N tokens match a cached prefix, you pay only for the new tokens. Cost saving: 50–80% on cached tokens. Latency saving: 200–400ms. Requirement: static portion must come first.
Combined Caching Flow
Request arrives │ ▼ Exact match cache check → hit: return cached response │ miss ▼ Semantic cache check → hit: return semantically matched response │ miss ▼ LLM call (with prompt cache prefix if applicable) │ ▼ Store in exact + semantic cache │ ▼ Return response
4.6 Fallback and Resilience Patterns
Fallback patterns
LLM APIs have uptime SLAs of 99.5–99.9% — meaning 4–44 hours of downtime per year. For business-critical systems, you need fallback architecture.
  • Primary → Secondary model fallback: if GPT-4o returns a 503, retry on Claude Sonnet. Configure at the model gateway layer so the calling application is unaware.
  • Graceful degradation: if the premium model is unavailable, fall back to a standard model with a simpler prompt. Quality degrades, but the system keeps running.
  • Cached fallback: if all live models are unavailable, return a cached response for semantically similar queries. Acceptable for FAQ answering; not for real-time data queries.
  • Circuit breaker: after N consecutive failures on a provider, open the circuit — stop sending requests and route all traffic to the fallback. After a timeout, probe the primary with one request; if it succeeds, close the circuit and resume normal routing.
  • Graceful degradation to rule-based: for the most critical path (e.g., order status lookup), have a simple rule-based fallback that doesn't use an LLM at all. Less quality, but zero dependency on LLM availability.
Enterprise Example
Design a Legal Document Q&A System

Your legal team handles 500 supplier contract reviews per month. Each review involves answering: "Does this contract include a limitation of liability clause? What is the notice period for termination? Does this align with our standard MSA?" Currently: a paralegal reads each contract manually — 4–6 hours per contract.

Layer 1 — Client/Surface
  • Interface: internal web interface for paralegals to upload contracts and ask questions
  • Auth: SSO via enterprise identity provider
  • Rate limit: 50 queries per user per hour
  • Validation: file type (PDF, DOCX), file size limit (50MB), virus scan before ingestion
Layer 2 — Orchestration
  • Query type classifier: "Is this a factual lookup or a comparison (does this match our MSA)?"
  • Factual lookups → single RAG retrieval + generation
  • Comparison queries → retrieve from both the uploaded contract and the MSA template, then compare
  • Multi-step queries → agent loop with multiple retrieval steps
Layer 3 — Model Gateway
  • Simple factual queries → Claude Sonnet (cost-effective, fast)
  • Complex comparison and multi-step → Claude Opus (better reasoning for legal nuance)
  • Routing classifier: keyword + query length heuristic (£0.001/query)
  • Fallback: if Claude unavailable → GPT-4o
Layer 4 — Knowledge/Retrieval
  • Per-contract index: each uploaded contract chunked (512 tokens, 50-token overlap) into a per-matter vector store namespace
  • Shared index: MSA template and standard clause library for comparison queries
  • Hybrid retrieval: semantic + BM25 keyword search (legal language has precise terms that semantic search sometimes misses)
  • Reranking: cross-encoder reranker applied to top 20–50 candidates before sending to LLM
Layer 5 — Data/Integration
  • Document storage: Azure Blob, organised by matter ID
  • Matter metadata: contract type, counterparty, date, assigned paralegal — from matter management system
  • Write-back: approved summaries and flagged clauses written back to matter record in document management system
Layer 6 — Observability/Governance
  • Every query and response logged with: matter ID, query, retrieved chunks, model used, response, latency, cost
  • PII handling: counterparty names and financial terms redacted in the LLM call, re-injected into the response after generation
  • LLM judge: 10% of responses evaluated for accuracy and groundedness
  • Audit trail: every AI-assisted response marked as "AI-assisted" in the matter record, including which document chunks were retrieved

8.1 The FRAME Design Framework

Use FRAME to ensure completeness when designing or presenting any AI system:

  • F — Functional requirements: who are the users, what tasks do they perform, what are the inputs and outputs, what is the volume and frequency?
  • R — Reliability and scale: what are the SLAs? What is expected volume now and in 12 months? Peak-to-average ratios? What happens when a component fails?
  • A — Architecture decisions: which of the five core design choices apply, and what did you choose and why?
  • M — Monitoring and evaluation: how do you know the system is working? What quality signals do you track? What are the alert thresholds? How does the feedback loop work?
  • E — Economics: what does this cost at current volume? At 10× volume? What are the cost drivers? What happens to unit economics as you scale?

8.2 High-Volume Hybrid: Recommendation + LLM

A common enterprise pattern: high-volume retrieval where LLM reasoning adds value but is too expensive to run on all candidates.

Stage 1 — Candidate retrieval (fast, cheap): collaborative filtering, semantic search, BM25 keyword — retrieves top-N candidates. Cost: < £0.001/query, < 50ms.

Stage 2 — LLM re-ranking and explanation (applied to top N only): LLM takes the top 10–20 candidates and applies reasoning — personalisation, explanation generation, filter application. Cost: £0.02–0.10/query.

Key design principle: never run the LLM on the full candidate space. Always use a cheap retrieval stage to filter to a manageable set first.

Two-Stage Hybrid Pattern
10M product catalogue │ ▼ Stage 1: Vector search + collaborative filter → top 20 candidates (< 50ms, < £0.001) │ ▼ Stage 2: LLM reranker on 20 candidates → top 5 with explanations (1,500ms, ~£0.05) │ ▼ Response to user: top 5 products with personalised explanations
8.3 Cost Math: Unit Economics at Scale

Before any AI system design is approved, you need the unit economics model:

Unit Economics Calculation
Daily cost = (queries/day) × (avg tokens/query) × (cost per 1K tokens) Example: customer service assistant - 50,000 queries/day - Average: 800 input tokens + 200 output tokens = 1,000 tokens - Model: Claude Sonnet at £1.50/1M input, £6.00/1M output - Input cost: 50,000 × 800 / 1,000,000 × £1.50 = £60/day - Output cost: 50,000 × 200 / 1,000,000 × £6.00 = £60/day - Total: £120/day = £3,600/month = £43,800/year

Key Takeaways
  • AI system design is about making the hard decisions before code is written. Model choice, routing strategy, caching approach, fallback architecture, multi-tenancy model — these decisions are expensive to reverse. Design time is when they cost nothing to change.
  • The 6-layer architecture separates concerns that must not be coupled. Surface, orchestration, model gateway, knowledge, data, and observability are each a distinct responsibility. Collapsing layers — especially coupling orchestration with model routing — creates systems that are expensive to change and impossible to govern centrally.
  • Cost is a design variable, not a post-launch discovery. Model tiering (routing simple queries to cheap models) typically saves 50–70% compared to using the most capable model for everything. Run the unit economics calculation before the system is built, not after the first invoice arrives.
  • Latency budget must be designed in, not hoped for. Work backward from the SLA: subtract network, subtract processing overhead, what's left is your LLM + retrieval budget. If the remaining budget is smaller than the P95 latency of your chosen model, you need a different model, a caching strategy, or a different SLA negotiation — not an optimistic assumption.
  • Fallback architecture is not optional for production systems. LLM API uptime is 99.5–99.9% — that's 4–44 hours of downtime per year. Design the primary → secondary model fallback, graceful degradation, and circuit breaker patterns before launch. Discovering you have no fallback during a production outage is the worst time to design one.
