What Is It (Plain English)
🎯 Explain Like I'm an Architect: Think of AI infrastructure the same way you think about database infrastructure. You don't need to understand the B-tree internals of PostgreSQL to make good database architecture decisions — but you do need to understand "this query will hit every row in a 500M-row table, that's going to be slow and expensive." AI infrastructure is the same: you don't need a PhD in CUDA programming, but you do need to know which decisions drive cost and latency, and which levers you actually control.
💡 What is a token? A token is the basic unit of text that LLMs process — and the unit that APIs charge for. Roughly 3–4 characters or 0.75 words in English. “Hello world” = 2 tokens. 1,000 tokens ≈ 750 words ≈ 1.5 pages. APIs charge separately for input tokens (your prompt) and output tokens (the model’s response). When you see pricing like “$2 per million tokens”, that is the cost per million of these text chunks.
AI infrastructure is the hardware, software, and serving stack that makes AI models run — fast enough, cheaply enough, and reliably enough to be useful in production.

Most architects never need to train a large model. But every architect deploying an AI system makes infrastructure decisions that directly affect cost, latency, and reliability:
  • Which model to use — a 7B parameter open-weight model self-hosted on your GPU, a 70B model via API, or a frontier model like GPT-4o at a premium price point
  • How to serve it — serverless API calls (pay per token), a dedicated endpoint (fixed cost), or self-hosted inference (capital cost + ops burden)
  • How to optimise it — quantisation (run a smaller, cheaper version with minimal quality loss), caching (avoid recomputing what you've already computed), batching (process multiple requests together for efficiency)
  • How to scale it — when traffic spikes, what happens? How does cost scale with volume? What's the latency at P95 under load?

Understanding AI infrastructure gives you the vocabulary and the mental models to have these conversations credibly — with engineering teams, cloud vendors, and finance stakeholders — without needing to have built a GPU cluster yourself.
Why Should I Care
💡 What is the context window? The context window is the maximum amount of text the model can hold in working memory at once — your prompt, conversation history, retrieved documents, and the model’s response all count toward this limit. Think of it as a whiteboard: everything the model needs to reason with must be written on this whiteboard. When it fills up, earlier content falls off and the model can no longer see it. Current models support 128K–1M tokens (roughly 90,000–750,000 words), but long agent workflows with rich tool outputs fill this faster than you expect.
For Solution Architects

Infrastructure decisions are cost decisions. The difference between a poorly chosen infrastructure setup and a well-optimised one can be 5–10× on the monthly AI bill at scale.

Concrete examples of infrastructure decisions that architects own:

  • Self-hosted vs API: a 7B model self-hosted on a single A10G GPU (£0.60/hr) can handle ~100 requests/minute at £0.001/request. The equivalent via GPT-4o API is £0.003–0.006/request — 3–6× more expensive, but zero ops burden, no GPU procurement, and automatic scaling.
  • Quantisation trade-off: a 70B model at FP16 precision requires ~140GB GPU memory (2× H100s). The same model quantised to 4-bit requires ~35GB (one H100). 75% memory reduction, 5–10% quality degradation. For many use cases, the quality loss is invisible; the cost saving is decisive.
  • Caching: if 15% of your API requests have identical system prompts, prompt caching saves 50–80% on those requests' input token costs. At 1M requests/day, this compounds quickly.
For Enterprise Architects

AI infrastructure introduces a new cost model to enterprise IT. Traditional software infrastructure costs are fixed or scale with compute (CPU/memory). AI inference costs scale with tokens — the amount of text processed. This is a fundamentally different cost structure:

  • Variable query cost: a 5-word query costs a fraction of a 5,000-word document analysis
  • Variable output: output length is model-determined — not fixed like a database row
  • Model swap leverage: a model swap (GPT-4o → GPT-4o-mini) can cut costs 10× with no infrastructure change
  • Context window as cost driver: context window length is the primary cost driver for long-document use cases — not the number of requests

Enterprise architects need to understand token economics to forecast AI costs accurately, build governance controls around them, and evaluate build-vs-buy decisions for AI serving infrastructure.

Think About It Like This
The Delivery Fleet Analogy

Imagine you're running a delivery operation. You need to get packages from warehouse to customers.

The GPU is your fleet. A high-end H100 GPU is like a fleet of 100 vans — enormous capacity, very fast, very expensive. An A10G is like 20 vans — still capable, much cheaper. A CPU trying to do inference is like a bicycle — technically possible, but not viable at any meaningful scale.

Quantisation is like load optimisation. Your vans can carry packages in custom foam packaging (full precision, FP16) — safe, but each van carries fewer packages. Or you can standardise packaging (quantise to INT4/FP4) — slightly less protection, but you fit 4× as many packages per van. The packages arrive in essentially the same condition; you've dramatically increased fleet utilisation.

Batching is filling the van before it leaves. Sending one package per van trip is wasteful. If you wait 50ms to collect 10 requests and process them together, each GPU computation amortises across 10 requests. Throughput goes up; cost per request goes down.

KV cache is your local depot. When a van makes a repeat delivery to the same neighbourhood (same prompt prefix), it's wasteful to drive from the central warehouse every time. The local depot stores the work already done for that neighbourhood — the van starts from there, not from scratch.

Speculative decoding is the fast courier service. A small motorbike (small draft model) guesses the next few deliveries and races ahead; a big van (the main model) quickly verifies and approves the guesses. When the guesses are right (they usually are), you've delivered 4 packages in the time it would take the van to deliver 1.

Serverless API vs self-hosted is the classic make-vs-buy decision: use a courier service (API) for variable demand — pay per package, no fleet management — or run your own fleet (self-hosted) for predictable high volume — higher fixed cost, lower per-package cost at scale.
Step-by-Step Walkthrough — The Core Concepts
4.1 GPU Memory: Why It's the Primary Constraint
🎯 Explain Like I'm an Architect: Why GPUs instead of CPUs for AI? A CPU is like a team of 10 world-class specialists — each one can handle any complex problem independently. A GPU is like a workforce of 10,000 labourers who can only do simple tasks, but can do them all simultaneously. AI model inference is basically millions of simple multiplications happening in parallel — the labourers win every time. A single H100 GPU has 16,896 CUDA cores working at once; your server CPU has 64 cores. For the specific arithmetic AI models need, the GPU is 100–1,000× faster. The cost? GPU memory (VRAM) is limited and expensive — an H100 has 80GB; a CPU server might have 1TB RAM at a fraction of the cost per GB. The entire AI infrastructure challenge is essentially: how do we fit our model into this expensive-but-fast VRAM?

Everything about LLM inference — which models you can run, at what speed, at what cost — is constrained by GPU memory (VRAM). The model weights must fit entirely in GPU VRAM before any inference can happen.

⚠ Common Misconception: "We can just use our existing cloud VMs with a lot of RAM for AI inference." Regular RAM (system memory) and GPU VRAM are not interchangeable. The AI model must live in GPU VRAM to run at useful speeds. Running a 7B model on CPU RAM instead of a GPU is technically possible but typically 10–50× slower — too slow for any production use case. Think of it like asking an accountant to do a warehouse worker's job: same building, completely wrong tool.

The KV Cache — Additional Memory on Top of Model Weights
🎯 Explain Like I'm an Architect: What is a KV cache? Imagine you're reading a long document and answering questions about it. Every time someone asks a new question, you don't re-read the entire document from scratch — you keep notes from your previous reading (the "KV cache") and just read the new question. That's exactly what the KV cache does: it stores the model's internal "notes" (key-value pairs from the attention mechanism) about the prompt it already processed. Without it, every new word generated would require re-reading the entire conversation from scratch — catastrophically slow. The trade-off: these notes consume GPU memory, and longer conversations mean bigger notes. A 128K-token conversation can have notes that are bigger than the model itself.

When generating a response, the model caches the key-value pairs from the attention computation for all processed tokens. The KV cache grows with sequence length. Longer inputs, more layers, and larger batches all multiply together. Context length is the most controllable lever architects have.

KV cache memory formula
KV cache memory = 2 × num_layers × num_heads × head_dim × seq_length × batch_size × bytes_per_value For Llama-3 70B at FP16, 4K sequence, batch size 1: ≈ 2 × 80 × 8 × 128 × 4096 × 1 × 2 bytes ≈ 1.3 GB

At 128K context length, the KV cache for a single request can approach 40GB on some models — exceeding the model weights themselves. This is why long-context serving is memory-constrained, not compute-constrained.

The Memory Bandwidth Bottleneck

LLM inference at typical batch sizes is memory-bandwidth bound, not compute bound. The GPU spends most of its time moving data between VRAM and compute cores — not doing arithmetic. This is why:

  • Bandwidth trumps FLOPS: a GPU with 2 TB/s memory bandwidth generates tokens faster than a GPU with 1 TB/s bandwidth, even at the same FLOP count
  • Smaller models often serve faster: less data to move per token, even when the larger model "fits"
  • Quantisation double benefit: helps not just by fitting the model in less memory, but by reducing data moved per operation
4.2 Quantisation: The Practical Decision Guide
🎯 Explain Like I'm an Architect: Quantisation is a compression trade-off that every architect makes decisions about — just usually in different domains. Think of it like image compression: a PNG file stores every pixel as a precise 24-bit colour value (full quality). A JPEG stores approximate colour values using fewer bits — the photo looks nearly identical, downloads 10× faster, and uses 10× less storage. Quantising an AI model is the same idea: instead of storing each weight as a precise 16-bit decimal number (FP16), you round it to a cruder 4-bit integer (INT4). The model is 4× smaller, loads 4× faster, uses 4× less GPU memory. The "quality loss" is usually 1–3% on benchmarks — often invisible in real applications. The architectural decision: how much quality loss is acceptable for your use case? Legal document review: probably FP16. Internal FAQ chatbot: probably INT4 is fine.

Quantisation reduces the numerical precision of model weights — from 16-bit floats (FP16) to 8-bit integers (INT8) or 4-bit integers (INT4/FP4). The model becomes smaller and faster; there is some quality degradation, which varies by model and task.

⚠ Common Misconception: "Quantisation is a lossy last resort — only use it if you have no other choice." In practice, 4-bit quantisation (AWQ, GPTQ) is the standard production choice for self-hosted open-weight models, not a compromise. At 4-bit, a 70B model outperforms most un-quantised 13B models. You get a much smarter model at a smaller footprint — the quality per GB of VRAM is often higher with a quantised larger model than an unquantised smaller one.

4.3 The Production Inference Stack
🎯 Explain Like I'm an Architect: "Model serving" vs "model training" — these are as different as a car factory and a motorway. Training is building the factory: you're teaching the model (takes weeks, costs millions, happens once). Serving (inference) is running finished cars on the motorway: you're using the already-trained model to respond to live user requests (happens continuously, must be fast and cheap). When you deploy an AI feature in your enterprise application, you are doing serving — not training. The "inference stack" (vLLM, SGLang, TGI) is the software that turns a pile of model weights into an API endpoint your application can call. Think of it as the web server for your AI model — just as Nginx turns your application code into a web service, vLLM turns your model weights into an AI service.

The inference stack is the software layer that takes model weights and serves them as an API endpoint. In 2026, the production standard is:

vLLM (most widely deployed)
  • PagedAttention: manages the KV cache like virtual memory in an OS — allocates KV cache in non-contiguous blocks ("pages"), eliminating fragmentation. Enables serving 2–4× more concurrent requests on the same GPU vs naïve KV cache management.
  • Continuous batching: instead of processing a batch as a unit (waiting for the slowest request), processes tokens continuously — new requests join the batch as soon as a slot is free. Dramatically improves GPU utilisation under variable load.
  • Prefix caching: caches KV states for common prompt prefixes — if 80% of requests share the same system prompt, that prefix is computed once and reused, cutting input processing time for those requests.
SGLang & TGI

SGLang — strong alternative for structured output and complex workflows:

  • Structured generation: runtime constraint enforcement during generation (JSON, constrained output)
  • Multi-call programs: state is shared across calls rather than re-initialised
  • Speculative decoding with RadixAttention

TGI (Text Generation Inference, Hugging Face) — good for: straightforward deployment of HuggingFace models; solid observability; simpler configuration than vLLM for standard use cases.

Batch vs Streaming Inference — The Latency SLA Decision
🎯 Explain Like I'm an Architect: Batch vs streaming inference is the same trade-off as batch ETL vs real-time event processing — and architects already understand this instinctively. Batch inference is like running a nightly SQL job: you queue up 10,000 documents, the model processes them overnight, you collect results in the morning. Great for throughput, irrelevant for latency. Streaming inference is like a real-time chat response: the model generates each word as it computes it, and your application displays it immediately (the "typing…" effect). Users perceive the response as instant even though full generation takes 5–10 seconds, because they see the first words in under a second. The architectural decision: what does your SLA require? A nightly report generation job needs batch. A customer-facing chatbot needs streaming. Getting this wrong means either unnecessary complexity (streaming for a batch job) or poor UX (waiting for the full response before displaying anything).
Batch Inference
  • Process many requests together; optimise for throughput
  • No real-time SLA — results delivered asynchronously
  • 50% cost discount at most API providers (async batch endpoints)
  • Use for: nightly document processing, bulk data enrichment, offline analytics, report generation
  • Latency: minutes to hours — irrelevant by design
Streaming Inference
  • Tokens sent to client as they are generated (Server-Sent Events / WebSockets)
  • Time-to-first-token (TTFT) matters — user sees response starting in < 1 second
  • Full generation latency (5–15 seconds for 300 tokens) felt less because response is progressive
  • Use for: customer-facing chatbots, copilots, real-time assistants
  • SLA metric: P95 TTFT < 1s; P95 generation speed > 15 tokens/second
⚠ Common Misconception: "Streaming inference is always better because users get faster responses." Streaming adds infrastructure complexity (persistent connections, SSE/WebSocket handling, partial response state management) and is unnecessary for back-end batch workloads. Use streaming only for interactive user-facing applications. A nightly document summarisation pipeline should use batch — it's simpler, cheaper, and processes more documents per hour.

Cold Start: Why Serverless Inference Has Latency Spikes
🎯 Explain Like I'm an Architect: "Cold start" in AI inference is identical to cold start in serverless functions (AWS Lambda, Azure Functions) — but 100× worse. When a Lambda function cold-starts, it takes 200–500ms to spin up the container. When an AI inference endpoint cold-starts, it takes 30–120 seconds to load 140GB of model weights into GPU VRAM. Imagine asking your database to load its entire data file into RAM before answering every query that comes in after an idle period. That's the cold start problem. For a customer-facing feature with sporadic traffic (e.g., an internal tool used 9–5 on weekdays), a serverless AI endpoint can introduce a 60-second first-request delay after any idle period — catastrophic for user experience. The fix: keep a "warm" instance running permanently (dedicated endpoint) — but that costs money even at zero traffic.
What Causes Cold Start
  • Model weights must be loaded from storage into GPU VRAM before any inference
  • A 7B model at FP16 = 14 GB to load. At ~2 GB/s typical storage throughput: ~7 seconds
  • A 70B model at FP16 = 140 GB to load: ~70 seconds
  • Container/runtime initialisation adds further overhead
  • Serverless providers scale to zero after idle periods — triggering cold start on next request
Cold Start Mitigation Strategies
  • Dedicated (always-on) endpoints: instance stays warm; no cold start; fixed cost even at zero traffic
  • Minimum instance count: keep at least 1 warm instance; scale to 0 never triggered
  • Provisioned concurrency: cloud-managed pre-warming (AWS Bedrock, Azure AI, Google Vertex)
  • Smaller quantised models: INT4 7B = 3.5 GB to load ≈ 2 seconds cold start vs 70 seconds for FP16 70B
  • Fast storage: NVMe SSD or RAM-backed model cache dramatically reduces load time
⚠ Common Misconception: "Serverless AI endpoints are always cheaper because you only pay when you use them." Serverless is cheaper only for intermittent workloads. For steady-state traffic, a dedicated instance runs at lower cost per request. More importantly: if your application has an SLA, cold start can violate it. Always model the traffic pattern before choosing serverless — the cost saving is real, but the cold-start penalty must be acceptable for your use case.
4.4 Speculative Decoding: Faster Generation for Free

LLM generation is slow because it's autoregressive — each token is generated one at a time, requiring a full forward pass of the model. A 70B model producing a 200-token response makes 200 sequential forward passes.

How It Works
  1. A small draft model (e.g. 7B) generates K tokens speculatively — very fast, it's small
  2. The large target model (e.g. 70B) verifies all K tokens in a single forward pass — parallel, not sequential
  3. If the draft's tokens match what the target model would have generated: accept all K tokens — you've generated K tokens in the time of 1 large model pass
  4. If the draft diverges at token j: accept tokens 1…j-1, reject j onwards, regenerate from j with the target model
Typical gain3–5× throughput improvement with zero quality degradation
RequirementsDraft model must use same tokeniser as target model; compatible token distributions
Best whenOutput is predictable — common phrases, structured formats, boilerplate code
Helps less whenOutput is highly creative or diverse — draft is wrong more often, rejection overhead increases
4.5 Disaggregated Prefill / Decode (P/D Split)
🟡 Optional depth — Relevant if you are planning infrastructure at 100+ GPU scale or overseeing a platform engineering team building an inference cluster. For most enterprise AI deployments using managed API services, this is background context rather than an immediate design decision. Safe to skim on a first pass.

LLM serving has two computationally distinct phases with completely different hardware requirements:

Prefill Phase

Processing the input prompt. Compute-intensive — all input tokens are processed in parallel. A 4,000-token input prompt is processed in one shot. This phase is compute-bound (arithmetic-heavy).

Decode Phase

Generating output tokens one at a time. Memory-bandwidth intensive — each generation step loads the full model weights and KV cache. This phase is memory-bandwidth bound.

Disaggregated serving flow
Request with long prompt (4K tokens) │ ▼ Prefill Pool (compute-optimised GPUs — e.g., H100 SXM) → Process input prompt in parallel → Generate initial KV cache → Transfer KV cache to decode pool │ ▼ Decode Pool (memory-bandwidth-optimised GPUs — e.g., H100 NVL) → Generate output tokens autoregressively → Using transferred KV cache
When it matters: disaggregated serving matters for large-scale deployments (100+ GPUs) or for use cases with highly variable input length (some requests with 100-token inputs, others with 100K-token inputs). For smaller deployments, the engineering complexity is not justified.
4.6 Multi-GPU Serving: Tensor Parallelism and Pipeline Parallelism
🎯 Explain Like I'm an Architect: When a model is too big for one GPU, you split it across multiple GPUs. There are two main ways, and the analogy is a factory production line vs a team of co-workers. Tensor parallelism is like a team of 4 co-workers all working on the same spreadsheet simultaneously — each handles different columns, they communicate every few seconds to share intermediate results, and together they finish in a quarter of the time. Fast, but they need to talk to each other constantly (high communication overhead). Pipeline parallelism is like a production line: GPU 1 does the first 20 layers, passes the result to GPU 2 which does layers 21–40, and so on. Less communication, but GPUs can sit idle waiting for their turn (pipeline "bubbles"). For enterprise architects overseeing a 70B model self-hosting project: tensor parallelism across 2–4 GPUs within a single server is the standard choice. Pipeline parallelism is for very large models spanning multiple servers.
4.7 Cost Model: Self-Hosted vs API
🎯 Explain Like I'm an Architect: The build/buy/rent decision for AI inference follows the same economics as every enterprise IT decision — but with a twist most architects miss. In traditional software, "build" means high upfront cost but low marginal cost at scale. For AI inference, "build" (self-hosted) means: buy or rent expensive GPUs, hire MLOps engineers to operate them, and manage model updates, security, availability, and capacity. "Buy" (API) means: pay per request, no ops, no GPU procurement, but vendor dependency and data-leaves-your-network concerns. The critical number most business cases miss is the ops overhead: running a self-hosted LLM endpoint requires roughly 0.25–0.5 FTE of MLOps engineering time. At a fully-loaded cost of £100K/year, that's £25,000–£50,000/year of hidden cost before you count a single GPU. Include this in every build-vs-buy analysis.
API Pricing (pay-per-token)
  • No fixed cost; scales to zero at zero usage
  • No GPU procurement, no ops burden
  • Cost scales linearly with usage
  • Subject to provider pricing changes and availability
Self-Hosted (fixed + variable)
  • Fixed cost: GPU reservation (£0.60–£5.00/GPU/hr depending on type)
  • Variable cost: electricity, networking (usually minor)
  • Requires MLOps team to operate
  • Fully predictable once utilisation is understood
Break-even analysis example
Self-hosted break-even daily requests: GPU cost: 1× A10G at £0.80/hr = £19.20/day Serves: ~100 req/min for a 7B model = 144,000 req/day max At 60% util: 86,400 req/day API equivalent (GPT-4o-mini at £0.0003/req): 86,400 × £0.0003 = £25.92/day Break-even: ~74,000 req/day Below 74K req/day → API cheaper (no ops overhead) Above 74K req/day → self-hosted cheaper

Architecture Perspective
The Infrastructure Decision Stack

Every AI system requires decisions at four levels:

Level 1 — Model Selection
Which model fits the quality/cost/latency envelope? Frontier API vs mid-tier API vs self-hosted open-weight.
Level 2 — Serving Strategy
API (serverless) vs dedicated endpoint vs self-hosted? Driven by demand pattern, data residency, ops capability.
Level 3 — Optimisation
Which efficiency techniques apply? Quantisation, caching, batching, speculative decoding.
Level 4 — Scaling
How does cost and latency behave as volume grows? Linear scaling? Break-even thresholds? Spot vs reserved?
Make all four decisions explicitly at design time. The default (always use the frontier API, no optimisations) is almost never the right answer for production systems.

Distributed Training (For Architects Who Oversee Fine-Tuning Projects)
🎯 Explain Like I'm an Architect: When a model is too big for one GPU during training, you need multiple GPUs working together. There are three main patterns — think of them as different ways to organise a large team project. Data parallelism: everyone gets a copy of the full project specification (model weights) and works on different data simultaneously, then merges notes at the end (gradient synchronisation). Works when the specification fits in one person's desk. Tensor parallelism: the project specification itself is split across desks — desk 1 has chapters 1–5, desk 2 has chapters 6–10. They need to constantly share results (high communication, like a Slack channel that never stops). Pipeline parallelism: desk 1 finishes their section and passes the work downstream, like a manufacturing production line. Less interruption, but some desks wait idle while the previous stage completes. Enterprise architects: for a typical fine-tuning project, data parallelism with QLoRA quantisation on 2–4 GPUs is almost always the right starting point.
Data Parallelism (DDP)

Copy the full model to N GPUs; each GPU processes a different batch; gradients are synchronised across GPUs. Straightforward — works when the model fits on one GPU. Most fine-tuning uses this.

Tensor Parallelism (TP)

Split individual weight matrices across N GPUs; each GPU holds a slice of every layer. Required when the model is too large for one GPU. High communication overhead between GPUs.

Pipeline Parallelism (PP)

Split model layers across N GPUs; GPU 1 runs layers 1–20, GPU 2 runs 21–40, etc. Lower communication overhead than TP; introduces pipeline "bubbles" (GPUs waiting). Used for very large models.

For a typical enterprise fine-tuning project (LoRA fine-tuning of a 7B–70B model): DDP with QLoRA (NF4 quantisation for fine-tuning) on 1–4 A100 80GB GPUs is the standard setup. Full pre-training requires hundreds to thousands of GPUs and is not an enterprise activity.
Check Yourself
Self-Assessment Questions
  1. Your team wants to self-host a Llama-3 70B model. A developer says one H100 80GB GPU should be sufficient. Explain whether this is correct — and what would actually happen if you tried to load the model.
    Answer: The developer is incorrect. At FP16, a 70B model requires 140 GB just for model weights (70B parameters × 2 bytes each). An H100 80GB has only 80 GB of VRAM — the model will physically not load, and inference cannot start. You need at minimum 2× H100 80 GB (160 GB total) for FP16, or you can quantise to AWQ/GPTQ 4-bit, which reduces memory to ~35 GB — fitting comfortably on a single H100 80 GB with room for the KV cache. For production with reasonable batch sizes, 2× A100 80 GB (quantised INT4/FP8) or 2× H100 80 GB (FP16) are the standard configurations for 70B models. The KV cache and framework overhead also consume additional VRAM on top of the model weights — so "just fits" on paper usually means "runs out of memory at non-trivial batch sizes."
    Simple explanation: 70B at FP16 needs 140 GB; one H100 only has 80 GB — it's like trying to fit a 140 GB file onto an 80 GB drive.
    Architecture takeaway: Always calculate model memory footprint before choosing hardware. The formula is simple: parameters × bytes-per-precision. Know it before every self-hosting conversation.
  2. Your nightly batch process sends 10,000 documents averaging 8,000 tokens each to an LLM for summarisation. The current API cost is £2,400/month. An engineer proposes switching to self-hosted. What questions must you answer before making this decision — and what are you likely missing from the cost model?
    Answer: First, verify the current cost: 10,000 docs × 8,000 tokens × 30 days = 2.4 billion tokens/month input, plus output. If that equals £2,400 the model pricing is quite low — worth checking whether async batch pricing (50% discount) has already been applied. For the self-hosted comparison: (1) GPU cost for a batch workload that runs overnight only — a spot/preemptible A100 at ~£1/hr for 4 hours/night = ~£120/month, dramatically cheaper than £2,400. But then add: (2) MLOps FTE overhead — 0.25 FTE at £100K/year = £25,000/year = £2,083/month. That immediately inverts the equation. (3) Storage, networking, monitoring, security, redundancy, model update effort. (4) The batch job currently has no SLA — if self-hosted GPU is unavailable one night, do you have a fallback? A complete cost model presents: (GPU cost) + (amortised ops FTE cost) + (amortised setup cost) + (risk premium for availability) vs API at current and projected volume.
    Simple explanation: The GPU is often the cheapest part of self-hosting — the people who operate it are usually the most expensive part.
    Architecture takeaway: Never present a self-hosted cost model with only GPU costs. Always include the fully-loaded ops overhead. Run the break-even analysis at 3 usage scenarios: current, 2× current, 5× current.
  3. What is prefix caching, and describe a concrete enterprise scenario where it produces the largest possible cost saving — with rough numbers.
    Answer: Prefix caching stores the KV cache computation for a static prefix shared across many requests — typically the system prompt and fixed few-shot examples. When a new request arrives with the same prefix, the model reuses the cached computation instead of reprocessing those tokens. The provider charges at a discounted rate (typically 50–90% off) for cached tokens. Largest saving scenario: a customer service agent with a 2,000-token system prompt handling 200,000 queries/day. Without prefix caching: 2,000 input tokens × 200,000 = 400M tokens/day of system prompt processing. With prefix caching at 75% discount: you pay 25% of £0.0015/1K = £0.000375/1K on those cached tokens. Over a month: saving ≈ 75% × 400M × 30 × £0.0015/1K = approximately £13,500/month — purely from caching a static system prompt. The saving is proportional to (system prompt length / average total request length) × request volume × input token price.
    Simple explanation: If you have a long fixed instruction at the start of every request, the model only needs to read it once instead of every time — and you pay a fraction of the price for those tokens.
    Architecture takeaway: Prefix caching is free money. Enable it first, before any infrastructure change. It requires zero hardware changes and typically delivers 20–50% input cost reduction for workloads with consistent system prompts.
  4. A colleague says your inference server is "compute-bound" and you need more powerful GPUs. You disagree — you believe it is memory-bandwidth-bound. How do you explain the difference, and what evidence would confirm your position?
    Answer: Compute-bound means the bottleneck is arithmetic operations — the GPU is doing maths as fast as it can and waiting for more to do. Memory-bandwidth-bound means the bottleneck is moving data between VRAM and compute cores — the GPU finishes its calculations quickly but spends most time waiting for the next chunk of data to arrive. LLM inference at typical serving batch sizes (1–8 concurrent requests) is almost always memory-bandwidth-bound. Evidence to confirm: check GPU utilisation metrics — if GPU compute utilisation is 30–50% but memory bandwidth utilisation is 80–95%, you are bandwidth-bound. The practical implication: upgrading to a GPU with higher FLOPS but the same memory bandwidth will not help — you need higher GB/s. This is why an H100 SXM (3.35 TB/s bandwidth) outperforms an H100 PCIe (2 TB/s) for inference even with similar compute specs. Quantisation helps throughput not just by reducing footprint but by reducing bytes moved per operation — a direct bandwidth benefit.
    Simple explanation: The GPU isn't struggling to do the maths — it's struggling to read the data fast enough. The bottleneck is the data highway, not the calculation engine.
    Architecture takeaway: When evaluating GPUs for LLM serving, prioritise memory bandwidth (GB/s) over raw FLOPS. For storage and retrieval-heavy workloads, this extends to NVMe speed and CPU-GPU transfer bandwidth.
  5. A business case for deploying a self-hosted Llama-3 70B model quotes GPU cost of £3.50/hr and expects to handle 50 requests/minute. You are presenting this to a finance stakeholder. What is incomplete and what would a complete cost model include?
    Answer: GPU hardware cost is only one of six components a finance stakeholder needs. The complete model requires: (1) GPU cost with redundancy — production needs at least 2 instances for high availability: £3.50 × 2 × 24 = £168/day. (2) MLOps engineering overhead — operating a self-hosted LLM requires monitoring, incident response, updates, capacity management — typically 0.25–0.5 FTE at £25,000–£50,000/year = £2,000–£4,000/month. (3) Storage — model weights (140 GB at FP16, 35 GB at INT4), log storage, monitoring data. (4) Networking — data transfer costs, especially cross-region. (5) Initial setup investment — vLLM configuration, load balancing, monitoring, CI/CD, security review: typically 2–4 weeks of engineering time (£15,000–£30,000 amortised). (6) Ongoing model refresh — migration and testing cost when a new model version is adopted. Compare the fully-loaded total against: API cost at current volume + projected volumes + the cost of the ops risk (GPU unavailability = your AI feature is down).
    Simple explanation: The GPU is the cheap part. The people managing it are the expensive part — and they are invisible in most business cases that only show hardware costs.
    Architecture takeaway: Present 3-year TCO for any self-hosting proposal: GPU cost (annualised) + ops FTE cost + setup amortisation + risk. Compare against 3-year API cost at projected volume. Let the numbers make the decision — not gut instinct.

8.1 PagedAttention: How vLLM Manages Memory

The KV cache is the primary memory consumer during inference. Naïvely, each request pre-allocates a contiguous block of memory equal to its maximum sequence length. This wastes memory (most requests are shorter than maximum) and causes fragmentation (free memory scattered in unusable gaps).

PagedAttention — virtual memory for the KV cache
  • Pages: memory is divided into fixed-size blocks of 16–32 tokens
  • On-demand allocation: each request is allocated pages on demand, not upfront
  • Non-contiguous: pages for a single sequence need not be contiguous in physical memory — a mapping table tracks which pages belong to which sequence
  • Immediate reclamation: when a sequence completes, its pages are immediately freed and available for other sequences
GPU memory utilisation~55% (naïve) → ~95% (PagedAttention)
Throughput impact2–4× more concurrent requests on the same GPU
8.2 Knowledge Distillation and Pruning
Knowledge Distillation

Train a small "student" model to mimic the output distribution of a large "teacher" model. The student learns not just the correct labels but the teacher's full probability distribution — which encodes more information than hard labels alone. GPT-4o-mini is (reportedly) a distilled version of GPT-4o.

For architects: distillation is relevant when you want to fine-tune a custom model for your domain that is small enough to self-host cheaply, but trained with the quality signal of a much larger model.

In a vendor conversation: if a vendor claims their 7B model performs comparably to a 70B model, ask: distilled from which teacher model? On which benchmark tasks? Evaluated against which baseline?

Structured Pruning

Systematically remove entire attention heads, neurons, or layers that contribute least to model quality. A pruned model has fewer parameters and is genuinely faster — unlike quantisation which keeps the same computation in lower precision.

For architects: pruning is less commonly needed in deployment decisions (quantisation usually suffices), but you may encounter it in model cards — e.g. "this model uses structured pruning to reduce from 70B to 40B parameters."

8.3 The Token Budget: A Financial Model

For any production AI system, build a token budget at design time:

Monthly token budget — worked example
Workload: customer service assistant Queries/day: 50,000 Input tokens/query: 1,200 (system prompt 800 + user message 400) Output tokens/query: 300 Days/month: 30 Monthly input tokens: 50,000 × 1,200 × 30 = 1,800,000,000 (1.8B) Monthly output tokens: 50,000 × 300 × 30 = 450,000,000 (450M) At Claude Sonnet pricing (£1.50/M input, £6.00/M output): Input cost: 1,800M × £1.50/M = £2,700/month Output cost: 450M × £6.00/M = £2,700/month Total: £5,400/month With prefix caching (800-token system prompt cached, 60% discount): Cached input: 800/1,200 × 1,800M × £0.60/M = £720/month Uncached input: 400/1,200 × 1,800M × £1.50/M = £900/month Output: £2,700/month Total with caching: £4,320/month (20% saving) With 10% semantic cache hit rate (avoids entire LLM call): Effective queries: 50,000 × 0.90 = 45,000/day Adjusted total: £4,320 × 0.90 = £3,888/month
Building this model before deployment surfaces the cost sensitivity to each variable. Output token length has 4× the price impact of input token length — so controlling output length (max_tokens parameter, concise prompting) has high leverage.
Key Takeaways
  • GPU VRAM is the primary constraint for LLM deployment — everything else follows from it. Model weights must fit in VRAM before any inference happens. A 70B model at FP16 needs 140 GB; quantised to INT4 it needs 35 GB. Know the memory requirement before choosing hardware or deciding between self-hosted and API.
  • Quantisation is the highest-leverage optimisation for self-hosted models. Going from FP16 to AWQ/GPTQ 4-bit reduces memory 4×, enabling larger models on smaller hardware with typically < 2% quality degradation on most tasks. Always evaluate on your specific task — benchmark averages may not represent your use case.
  • LLM inference is memory-bandwidth-bound, not compute-bound. The bottleneck is moving data between VRAM and compute cores, not arithmetic. Memory bandwidth (GB/s) matters more than FLOPS for serving throughput, and quantisation helps throughput not just by reducing size but by reducing bytes moved per operation.
  • The self-hosted vs API decision is a volume and ops-capability threshold. Below ~50–100K requests/day for a mid-tier model, API is almost always cheaper once ops overhead is included. Above that threshold — or with strict data residency requirements — self-hosted pays off. Run the break-even analysis with your actual numbers, not generic estimates.
  • Cold start latency is a hidden SLA risk for serverless inference. Loading a 70B model from storage into GPU VRAM takes 60–120 seconds. For customer-facing features or any SLA-bound workload, always provision a minimum warm instance — the cost is trivial compared to the UX damage of a 90-second first-response delay.
  • Batch inference is 50% cheaper and simpler than streaming for non-interactive workloads. Nightly document processing, bulk enrichment, and report generation should use async batch endpoints. Reserve streaming for interactive user-facing applications. Getting this right halves the cost of non-real-time AI workloads.
  • Caching (prefix, semantic, KV) is cost reduction that requires no hardware change. Prefix caching saves 50–80% on static system prompt tokens. Semantic caching eliminates entire LLM calls for repeated similar queries. Prompt caching is available from most API providers at 50–90% discount on cached tokens. Implement caching before considering infrastructure changes — it typically delivers 20–40% cost reduction with minimal engineering effort.
